# Allowlist for the `upgrade-paths` sub-gate (spec 007 FR-003, extended
# by spec 008 FR-004). Each non-comment non-empty line names a pair
# whose scaffold → upgrade → verify round-trip is documented-broken
# against the current candidate. The gate still attempts the round-trip
# and logs the result, but a fail against an allowlisted pair does NOT
# block the sub-gate.
#
# Maintenance: add a row only when (a) the failure mode is understood and
# documented, and (b) fixing it is out of scope for the current rc cut.
# Delete rows as their upgrade paths are fixed.
#
# Format: one entry per line; comments start with `#`. Two row shapes:
#   <tag>             — allowlist every variant of this source-rc (back-
#                       compat with spec-007's tag-only allowlist).
#   <tag>:<variant>   — allowlist just this (source-rc, variant) pair.
# Each entry SHOULD be annotated with a follow-up issue or commit
# reference so reviewers can trace the decision.

# v1.0.0-rc3 — frozen-tag self-bootstrap defect: at rc3
# scripts/pre-release-gate.sh line 188 executed `exec bash "$0" "$@"` AFTER
# a prior `shift 2` had already consumed `--target <tag>` from the positional
# argv. The re-exec therefore ran the candidate orchestrator with no
# `--target` flag, which causes the self-bootstrap to auto-select the
# highest pre-release tag on the branch (rc11) instead of staying on rc3.
# The resulting "candidate at HEAD vs candidate via re-exec" mismatch
# trips the upgrade-paths round-trip even though rc3's own logic was
# fine at the time it was cut. Fixed in rc4 by saving the original argv
# into `original_args=("$@")` before the `shift` and re-execing with
# `exec bash "$0" "${original_args[@]}"`. Allowlisted because the defect
# lives in the frozen rc3 tag and cannot be retro-patched; the fix
# shipped in rc4 and forward.
v1.0.0-rc3
