## Task Decomposition Guidance for Next `/speckit-tasks`

Future task generation MUST use these rules:

- Each implementation task touches one coherent behavior, disposition, or lifecycle transition.
- Each task cites exactly one primary verification command/test in the task text.
- Secondary checks may be mentioned only as review context, not as the primary verification command.
- Cluster work must be split into sequential atomic subtasks unless one named primary test verifies the whole cluster end-to-end.
- A multi-issue PR is allowed only when the issues share one behavior and one primary verification command/test verifies the whole cluster.
- Close-only tasks (`wontfix-and-close`, `defer-to-v2`, `consolidate-with-other-issue`, `close-as-duplicate`) use one primary verification command such as `gh issue view <N> --json state,labels,comments`.
- Fix-and-close tasks use one primary verification command from the existing harness, selected to match the changed behavior, then route code-reviewer review as the review gate.
- Dispatch-log updates are not bundled into unrelated implementation tasks unless the same primary verification command proves the lifecycle transition.
- If a proposed task needs two independent primary commands to prove correctness, split it into two tasks.
- If two tasks could merge independently without breaking each other, generate two tasks with separate verification commands.
