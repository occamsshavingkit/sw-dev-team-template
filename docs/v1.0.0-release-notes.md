# v1.0.0 Release Notes

Operator note for G-9 rollback and upgrade handling.

## Upgrade path

- For downstream projects on `v1.0.0-rc15`, upgrade to `v1.0.0` after running `scripts/upgrade.sh --dry-run` first.
- Review the dry-run diff before applying the upgrade.
- If the dry-run reports conflicts, stop and resolve them manually; keep local customizations, adopt upstream changes only where intended, then rerun the dry-run.
- After any upgrade, rerun the smoke tests and the issue-feedback onboarding check.

## Rollback targets

- Primary rollback target: `v1.0.0-rc15`.
- Secondary rollback target: `v1.0.0-rc14`.
- Tertiary rollback target: the previous known-good stable tag in the downstream project history.

## rc3 workaround boundary

- Issue #84 does not rewrite `v1.0.0-rc3` in place.
- For an already-affected rc3-era tree, inspect the worktree diff, commit only after review or restore from VCS, then use current `v1.0.0-rc15` `scripts/upgrade.sh --dry-run` from a clean branch or worktree for later previews.
- Issue #106 does not rewrite `v1.0.0-rc4` in place. The rc4
  `scripts/upgrade.sh --dry-run` helper-write behavior is historical
  and immutable. If an rc4-era dry run modifies `scripts/upgrade.sh`,
  `scripts/lib/manifest.sh`, or `scripts/lib/first-actions.sh`, restore
  those files from VCS before applying any upgrade, then rerun the dry
  run using the current rc15 candidate from a clean branch or worktree.

## Local agent supplements

- `.claude/agents/*-local.md` files are project-specific supplements and survive upgrades unless the local project intentionally changes them.
- Keep local agent routing and other project-specific supplement behavior intact when reviewing upgrade conflicts.

## Release-audit scope

- Classify each release/version item as one of: downstream product artifact, project-filled template register, or upstream framework/template artifact.
- Downstream product release work may inspect framework artifacts only to classify scope.
- Do not edit `TEMPLATE_VERSION`, shipped agent contracts, rc stabilization docs, scaffold / upgrade scripts, or other framework-managed files during a product-only release audit.
- If a framework defect is found, file it upstream instead of patching it locally unless the current task explicitly authorizes framework maintenance.

## GitHub Release object

- The GitHub Release object starts at `v1.0.0` final.
- Do not create a GitHub Release object for `v1.0.0-rc15`.

## Known issues in this release

### VERSION file stamp mismatch

The `v1.0.0` tag was cut without first bumping the `VERSION` file from
`v1.0.0-rc15` to `v1.0.0`. The published tag carries `VERSION=v1.0.0-rc15`
in the tagged tree.

This tag is **intentionally left immutable** (customer ruling 2026-05-27).
It is a cosmetic defect: `scripts/upgrade.sh` stamps downstream projects'
`TEMPLATE_VERSION` with the tag name `v1.0.0`, not the VERSION file
content, so upgrade-path behaviour is unaffected.

Forward correction: the `version-stamp` precondition sub-gate has been
added to `scripts/pre-release-gate.sh` to enforce `VERSION == tag name`
before any future release tag is cut. See `docs/versioning.md §
Known issues` for the full incident record and the PM lessons entry
recommendation.
