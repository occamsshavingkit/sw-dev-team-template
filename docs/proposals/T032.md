## 3. Implementation sketch

- **Module / file set touched:** `docs/model-routing-guidelines.md` only.
- **Public interfaces introduced or modified:** no code API; document routing behavior for frontier escalation and provider/model fallback.
- **Key internal data shapes:** requested tier, requested effort, fallback tier/model, fallback reason, escalation trigger, role authority, review obligation.
- **Control flow in guidance:** add fallback behavior after elevation/lowering rules or governance; define fallback as execution substitution only, with unresolved conflicts or repeated failed fallback returning to `tech-lead` for routing.
- **Escalation rule:** safety, security, privacy, API, data-model, release-blocking, architecture, and unresolved-disagreement work escalates to `frontier`/`xhigh` rather than silently downgrading.
- **Non-goal:** do not define the exact fallback log format; T033 covers required log fields and authority-preservation record shape.

## 6. Risks and mitigations

|---|---|

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** Fallback guidance can fail in production if it allows provider/model substitution during safety, security, privacy, API, data-model, or release-blocking work without an explicit stop condition; a weaker model could silently handle frontier-required work and produce an unreviewed high-impact decision.
- **F-2 [Major]:** The proposal says repeated failed fallback returns to `tech-lead`, but does not define what counts as repeated or failed; implementations may keep cycling providers until context is stale, cost is high, or the original escalation rationale is lost.
- **F-3 [Major]:** If fallback is documented only as execution substitution, but not tied to preserving the original requested tier and effort in evidence, reviewers cannot later tell whether a downgrade occurred or whether the fallback stayed within the authorized risk envelope.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 closed:** Fallback guidance blocks silent downgrade for safety, security, privacy, API, data-model, release-blocking, architecture, and unresolved-disagreement work.
- **F-2 closed:** The second failed provider/model attempt for the same task is now a stop condition requiring return to `tech-lead`.
- **F-3 closed:** Fallback evidence now records requested tier and requested effort so reviewers can detect downgrades.
