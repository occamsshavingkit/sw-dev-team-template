## 3. Implementation sketch

- **Module / file set touched:** `docs/model-routing-guidelines.md` only.
- **Public interfaces introduced or modified:** none; this clarifies binding role-routing semantics and release verification expectations.
- **Key internal data shapes:** binding routing policy, provider-neutral capability tiers, exact provider model IDs, `verified on` date, and release-time verification evidence.
- **Control flow in guidance:** replace draft-status wording with binding-policy wording for role tier, effort, and plan-mode selection; keep provider model IDs as mappings that must be reverified before release.
- **Verification rule:** require release-time verification of exact model IDs and aliases before tagged release; any pinned provider ID in shipped files must carry or cite a current verified-on date.
- **Authority rule:** preserve existing wording that effort level changes reasoning spend, not role authority or source-of-truth hierarchy.
- **Non-goal:** do not add new provider classes or change tier defaults; later M5 tasks cover cross-AI routing expansion.

## 4. Dependencies touched

- **Libraries added or bumped:** none.
- **Internal modules consumed:** existing `docs/model-routing-guidelines.md` policy, effort, Codex effort, plan-mode, role default, and governance sections.
- **External services / APIs:** OpenAI and Claude provider docs already cited by `docs/model-routing-guidelines.md`; no new external dependency.

## 5. Test plan outline

- **Governance check:** verify pinned model IDs in shipped files require a verified-on date or release evidence.
- **Prior-art coverage:** verify the wording implements `docs/prior-art/T027.md` without hard-pinning every provider ID.
- **Regression command:** run `git diff --check`.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Binding wording accidentally locks downstream projects to one provider. | Bind routing semantics and verification cadence, not a mandatory vendor. |
| Exact model IDs become stale after release. | Require release-time verification and verified-on evidence for pinned IDs. |
| Effort guidance is mistaken for authority. | Preserve the explicit authority hierarchy already in the policy. |
| M5 provider-expansion work is pulled into M4. | Keep this task to binding status and release verification only. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **Critical:** If the status change makes the whole document binding
  without separating provider-neutral policy from provider-specific
  mappings, downstream projects can treat transient model IDs as mandatory
  vendor lock-in.
- **Major:** If "verified on" evidence is required but no acceptable
  evidence location is named, release reviewers can pass stale IDs with an
  untraceable date string that cannot be audited later.
- **Major:** If aliases and exact model IDs are not both covered by the
  release-time check, a shipped alias can silently drift to a different
  model while the pinned-ID rows appear compliant.

### 8.2 Rebuttals / revisions (`software-engineer`)

- Addressed: `docs/model-routing-guidelines.md` binds provider-neutral
  routing semantics while keeping provider-specific mappings advisory
  until release verification.
- Addressed: acceptable release evidence locations are named: release
  notes, `docs/pm/LESSONS.md`, or a linked release-review artifact with
  a `verified on` date.
- Addressed: release-time checks cover both exact model IDs and aliases.
