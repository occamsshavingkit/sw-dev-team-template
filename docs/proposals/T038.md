## 3. Implementation sketch

- **Module / file set touched:** create `schemas/generated-artifact.schema.json`; do not modify generated runtime files in this task.
- **Public interfaces introduced or modified:** new JSON Schema 2020-12 validation contract for generated-artifact manifests consumed by T039/T040 and G6 checks.
- **Key internal data shapes:** manifest record with `artifact_type`, `authority`, `canonical_inputs`, `generator`, `generated_at`, `output_path`, `checksum`, `review_gate`, `retention`, and optional `line_count` / `token_count` evidence.
- **Authority values:** require generated artifacts to declare non-canonical authority such as `generated` or `candidate`; reject records that promote generated output to canonical policy without an explicit promotion path.
- **Freshness fields:** require enough provenance to reproduce or detect drift: generator command/version, canonical input list, output path, and checksum.
- **Control flow in broad strokes:** T040 writes generated runtime candidates and a manifest shaped by this schema; T039 or G6 validation checks schema validity and drift evidence.

## 5. Test plan outline

- **Schema self-check:** validate `schemas/generated-artifact.schema.json` as draft 2020-12.
- **Positive fixtures:** accept a runtime-agent candidate manifest that cites canonical inputs, generator command, checksum, review gate, retention, and line-count evidence.
- **Negative fixtures:** reject missing canonical inputs, missing checksum, canonical authority claims for generated files, absent review gates, and malformed output paths.
- **Integration prep:** ensure T040 can emit a manifest without introducing a new source hierarchy.
- **Regression command:** run `git diff --check` and the eventual T039/T040 validation commands once implemented.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Schema validates generated prose instead of provenance. | Model manifests only, not full Markdown semantics. |
| Generated files become manual mirrors of policy. | Require canonical inputs, generator metadata, and generated/candidate authority. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1:** Manifest records can cite incomplete canonical inputs — **How it fails:** generated artifacts pass with only one input path even when the compiler used `CLAUDE.md`, `AGENTS.md`, role files, and local supplements, so freshness checks miss source drift. — **Severity if unaddressed:** blocker.
- **F-2:** Checksums are not bound to an algorithm or output bytes — **How it fails:** arbitrary checksum strings validate, letting a hand-edited generated file keep a stale or fake freshness marker. — **Severity if unaddressed:** major.
- **F-3:** Authority values allow promotion by wording loophole — **How it fails:** records avoid the literal `canonical` value but still use labels such as `authoritative` or `policy`, causing downstream consumers to treat generated output as source authority. — **Severity if unaddressed:** blocker.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 addressed:** `schemas/generated-artifact.schema.json` requires manifest-level and artifact-level `canonical_inputs`, including `CLAUDE.md`, `AGENTS.md`, and `.claude/agents/*.md` entries; T040 emits role-specific inputs and local supplements when present.
- **F-2 addressed:** each artifact checksum is an object with `algorithm: sha256` and a 64-character lowercase hex digest over the generated output bytes.
- **F-3 addressed:** authority values are limited to `generated`, `candidate`, or `generated-candidate`; canonical/policy/authoritative labels are rejected.
