## 8. Duel

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** If label names and category descriptions do not map one-to-one to the required issue body fields, triage can classify a report as a framework gap while the body still omits version, location, observed outcome, or redaction evidence; the failure is an upstream issue that appears routable but cannot be reproduced or safely filed.
- **F-2 [Major]:** If the taxonomy text does not state that labels are repository-scoped and must be created or mapped per downstream repository, downstream maintainers can assume labels propagate automatically; the failure is misclassified or unlabeled issues silently bypassing the intended framework-gap routing.
- **F-3 [Critical]:** If label examples include concrete downstream identifiers or invite product-specific category names, the taxonomy can undermine Rule 0 redaction; the failure is leakage of customer/project identity through the classification field even when the issue body is sanitized.

### 8.2 Rebuttals / revisions (`software-engineer`)

- F-1 addressed: `docs/ISSUE_FILING.md` states labels complement, not replace, the required issue body fields and maps each label to the relevant filing field.
- F-2 addressed: the taxonomy states GitHub labels are repository-scoped and downstream repositories must create or map labels before relying on label triage.
- F-3 addressed: the taxonomy prohibits customer names, downstream repository names, product codenames, site identifiers, and private facts in labels; examples remain abstract framework labels only.
