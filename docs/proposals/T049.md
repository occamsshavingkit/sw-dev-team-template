# Proposal — T049

## 1. Task reference

- **Task ID:** T049
- **One-line task statement:** Add gated manual or scheduled improvement workflow in `.github/workflows/improve-template.yml` with PR-only behavior and safe no-op or summary output on failure.
- **Trigger clauses that fire:** 3 — cross-module boundary.
- **Prior-art reference:** `docs/prior-art/T049.md`.
- **ADR reference:** none.

## 2. Chosen ADR path

- **Alternative chosen:** none — no ADR.
- **Reason:** The task implements the M7 workflow policy directly in GitHub Actions; no ADR artifact exists.

## 3. Implementation sketch

- Create `.github/workflows/improve-template.yml` and parent directories if absent.
- Define both triggers, but keep manual-first behavior:
  - `workflow_dispatch` with one required `target` input naming exactly one GitHub issue number, issue URL, or framework-gap identifier, plus optional `dry_run` input defaulting to `true`.
  - `schedule` at low frequency, initially weekly, with every scheduled run exiting as a no-op unless repository variable `ENABLE_M7_SCHEDULED_IMPROVEMENTS` is exactly `true` and `M7_SCHEDULED_TARGET` names one target.
- Set minimal permissions: read contents by default; use a separate PR-creation job with `contents: write` and `pull-requests: write` only after preflight, generation, validation, and T050 patch-size checks pass. Add `issues: write` only to a reporting job if issue comments are implemented.
- Run contract checks before creating a branch or PR.
- Generate at most one bounded change for the selected issue/gap.
- If checks, generation, validation, or size checks fail, emit a GitHub Step Summary and do not create or update a branch or PR. Keep reporting summary-only unless issue comment creation is explicitly implemented with a matching permission need.
- Prohibit automation from approving, merging, or pushing directly to the default branch.

## 4. Dependencies touched

- No package dependencies unless implementation chooses an approved first-party GitHub CLI already present on runners.
- Internal dependencies: `docs/workflow-pipeline.md`, contract-check scripts, future T050 patch-size limits, T051 review sequence.
- External service: GitHub Actions, GitHub PRs/issues, per `docs/prior-art/T049.md`.

## 5. Test plan outline

- Unit: YAML syntax validation and shell syntax validation for inline scripts.
- Integration: dry-run workflow path with a dummy issue input, verifying checks run before branch/PR creation.
- System / acceptance: confirm successful path creates a PR only and failure path creates no default-branch commit.
- Performance / security: verify least-privilege permissions, no secrets exposure, no self-approval, and no `pull_request_target` use.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Workflow gains excessive write permissions. | Split read-only preflight from write-capable PR/report jobs and avoid write access until checks pass. |
| Scheduled runs create surprise PRs. | Manual-first design; schedule is no-op unless explicitly enabled. |
| Failure leaves a half-created branch or PR. | Order steps so checks and patch-size validation happen before PR creation where possible. |

## 7. Open questions

- None for code start. Decision: include the scheduled trigger now, but make it a no-op unless explicitly enabled by repository variables and still limited to one configured target.

## 8. Duel

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** If branch or PR creation happens before contract checks and T050 size validation, a failing run can leave a bot branch or half-formed PR that maintainers mistake for reviewable output; the failure is unsafe automation state instead of the promised no-op/failure report.
- **F-2 [Critical]:** If the same job has `contents: write`, `pull-requests: write`, and any path to approve, merge, or update after review, automation can bypass human review boundaries; the failure is a self-authoring/self-advancing PR contrary to FR-014 and Hard Rule #3.
- **F-3 [Major]:** If scheduled runs rely on unset or loosely parsed variables, a default-branch schedule can select stale or multiple targets; the failure is surprise recurring PR creation outside the one-improvement-at-a-time model.

### 8.2 Rebuttals / revisions (`software-engineer`)

- F-1 addressed: `.github/workflows/improve-template.yml` runs contract checks, generation, and T050 patch-size enforcement in the read-only preflight job before any PR-creation job can run.
- F-2 addressed: PR creation is isolated in a later job with `contents: write` and `pull-requests: write`; the workflow does not approve, merge, enable auto-merge, or use `pull_request_target`.
- F-3 addressed: scheduled runs no-op unless `ENABLE_M7_SCHEDULED_IMPROVEMENTS` is exactly `true` and `M7_SCHEDULED_TARGET` names one target; target parsing rejects blank, comma-separated, or whitespace-separated targets.
