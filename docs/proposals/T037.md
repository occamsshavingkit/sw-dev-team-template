## 2. Chosen ADR path

## 3. Implementation sketch

- **Module / file set touched:** create `schemas/model-routing.schema.json`; do not edit `docs/model-routing-guidelines.md` unless a later review finds policy/schema mismatch.
- **Public interfaces introduced or modified:** new JSON Schema 2020-12 validation contract for routing entries and fallback events.
- **Key internal data shapes:** `routing_entry` records with `role`, `tier`, `effort`, `model`, optional `verified_on`, and optional `notes`; `fallback_event` records with task, agent, requested/actual routing, reason, timestamp, and `authority_preserved`.
- **Stable enums:** constrain `tier` to `fast`, `standard`, `strong`, `frontier`; constrain `effort` to `low`, `medium`, `high`, `xhigh`, `max`; constrain `fallback_reason` to values already listed in `docs/model-routing-guidelines.md`.
- **Provider model handling:** allow provider/model strings and `*-class` labels; require `verified_on` only when a release pins exact provider IDs.
- **Authority rule:** validation of routing metadata does not change role ownership, review gates, or customer-interface constraints.

## 4. Dependencies touched

- **Libraries added or bumped:** none in this task; `docs/prior-art/T037.md` names JSON Schema 2020-12 and Ajv/ajv-cli candidates for later validation.
- **Internal modules consumed:** `docs/model-routing-guidelines.md`, `docs/prior-art/T037.md`, and T039 lint integration.
- **External services / APIs:** none; provider catalogs are release-verification evidence, not schema dependencies.

## 5. Test plan outline

- **Schema self-check:** validate `schemas/model-routing.schema.json` as draft 2020-12.
- **Positive fixtures:** accept representative role routing entries, provider-class model strings, exact model IDs with `verified_on`, and the documented fallback event example.
- **Negative fixtures:** reject unknown tiers, unknown effort values, unknown fallback reasons, fallback events lacking `authority_preserved`, and exact pinned model IDs without verification evidence.
- **Policy trace check:** compare schema enums against `docs/model-routing-guidelines.md` values before G6 acceptance.
- **Regression command:** run `git diff --check` and the eventual T039 lint command once implemented.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Schema hard-codes stale provider model IDs. | Validate string shape and verification evidence, not provider catalogs. |
| Fallback records pass without preserving authority. | Require a concrete `authority_preserved` assertion on fallback events. |
| Routing schema drifts from policy prose. | Keep stable enums copied from `docs/model-routing-guidelines.md` and add a policy trace check. |
| Implementation changes binding routing semantics instead of validating them. | Stop and route an ADR at `docs/adr/fw-adr-0009-m6-runtime-contract-pipeline.md` before changing default tiers, provider requirements, source authority, persistence, serialization, or public boundaries. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1:** Exact provider IDs pass without release verification — **How it fails:** a stale concrete model string is accepted as a routing entry without `verified_on`, so release evidence can route specialists to unavailable or deprecated models. — **Severity if unaddressed:** major.
- **F-2:** Fallback events preserve syntax but not authority — **How it fails:** `authority_preserved` can be a bare boolean with no evidence of which gate, role owner, or customer-interface rule was preserved. — **Severity if unaddressed:** blocker.
- **F-3:** Policy enum drift is not caught — **How it fails:** schema enums for `tier`, `effort`, or fallback reasons silently diverge from `docs/model-routing-guidelines.md`, causing valid policy records to fail or invalid routing to pass. — **Severity if unaddressed:** major.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 addressed:** `schemas/model-routing.schema.json` permits provider-class labels without verification but requires `verified_on` for non-`*-class` exact model strings.
- **F-2 addressed:** fallback events require `authority_preserved` as a concrete string assertion matching role/output/review/customer-interface preservation, not a boolean.
- **F-3 addressed:** stable policy values for `tier`, `effort`, and `fallback_reason` are encoded as schema enums copied from `docs/model-routing-guidelines.md`, and T039 validates a representative sample against the schema.
