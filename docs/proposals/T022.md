## 3. Implementation sketch

- **Module / file set touched:** `docs/TEMPLATE_UPGRADE.md` for guidance; no production script edits in this proposal unless `tech-lead` expands T022 beyond the documented task statement.
- **Public interfaces introduced or modified:** documented operator guidance that missing `docs/intake-log.md` is created from `docs/templates/intake-log-template.md`; existing logs must be preserved.
- **Key internal data shapes:** live intake log path `docs/intake-log.md`, source template path `docs/templates/intake-log-template.md`, and permanent customization path `.template-customizations`.
- **Control flow in guidance:** tell maintainers to dry-run upgrade/repair first, create `docs/intake-log.md` only when absent, seed it from the intake-log template, and add it to `.template-customizations`; if the file already exists, leave contents untouched.
- **Upgrade distinction:** normal upgrade may add a missing upstream-shipped file, but project-owned intake history must not be overwritten; guidance should explicitly prefer preserve-overwrite for existing logs.
- **Repair distinction:** repair guidance may recreate missing register files, but must warn that destructive repair paths require backups or dry-runs before touching append-only project records.
- **Non-goal:** do not invent historical intake entries for old projects; a missing log starts empty and future customer questions are recorded from that point forward.

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** If `docs/TEMPLATE_UPGRADE.md` documents missing-log creation but does not name the actual repair or upgrade step that performs it, downstream maintainers can follow the guide and still finish with no live `docs/intake-log.md`, leaving `tech-lead` without the required append-only customer-question register.
- **F-2 [Critical]:** If the guidance says to recreate register files without an explicit absent-vs-existing branch and backup/dry-run ordering, an operator can overwrite an existing intake log during repair, permanently losing customer-question history that should be preserved as project-owned evidence.
- **F-3 [Major]:** If the guidance treats `docs/intake-log-ARCHIVE.md` or the template file as satisfying the live-register requirement, upgraded projects can appear compliant in documentation review while runtime workflows still lack writable `docs/intake-log.md` for new session entries.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 revision:** `docs/TEMPLATE_UPGRADE.md` now names the actual
  paths: normal upgrade uses `scripts/upgrade.sh --dry-run` followed by
  `scripts/upgrade.sh`, whose missing-file add path creates absent
  upstream-shipped `docs/intake-log.md`; destructive in-place repair uses
  an explicit operator step after `scripts/repair-in-place.sh --dry-run`
  to create the absent live log from `docs/templates/intake-log-template.md`.
- **F-2 revision:** The guidance branches on absent versus existing
  `docs/intake-log.md`, requires dry-run before upgrade or repair,
  requires a backup before destructive repair, and states existing logs
  must not be overwritten, truncated, or replaced.
- **F-3 revision:** The guidance states that `docs/intake-log-ARCHIVE.md`
  and `docs/templates/intake-log-template.md` do not satisfy the live-log
  requirement, and that `.template-customizations` preserves project-owned
  intake history on future upgrades.
