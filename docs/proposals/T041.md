## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1:** Cases verify previous generated output instead of canonical role contracts — **How it fails:** a regression passes because runtime candidates match old generated prose even after `CLAUDE.md`, `AGENTS.md`, or `.claude/agents/*.md` changed. — **Severity if unaddressed:** blocker.
- **F-2:** Core-role coverage omits negative forbidden behaviors — **How it fails:** cases list expected behavior but do not fail when `researcher` contacts the customer, `software-engineer` steals QA work, or `qa-engineer` accepts summary-only test evidence. — **Severity if unaddressed:** major.
- **F-3:** Pass/fail criteria depend on subjective prompt quality — **How it fails:** reviewers cannot reproduce results because criteria such as "good answer" or "reasonable escalation" replace concrete invariants and forbidden behaviors. — **Severity if unaddressed:** major.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 addressed:** `docs/prompt-regression.md` states that canonical sources are the oracle and previous generated output is evidence only, never authority.
- **F-2 addressed:** every core case includes explicit forbidden behaviors, including direct customer contact, role-stealing, summary-only QA acceptance, safety-critical approval without authorization, and generated-authority inversion.
- **F-3 addressed:** each case uses concrete expected invariants, forbidden behaviors, evidence paths, and pass/fail criteria rather than subjective prompt quality.
