## 8. Duel

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** If the one-improvement rule does not define a measurable target boundary, automation can group several related gaps into one branch and PR; the failure is an oversized change that bypasses the intended review and rollback granularity.
- **F-2 [Major]:** If failure outputs are described only as "no-op status" without naming what is reported and where, failed runs can disappear without a branch, PR, issue comment, or actionable summary; the failure is silent automation loss with no reproducible diagnostic trail.
- **F-3 [Major]:** If the documentation cross-references T049-T051 without stating the interim binding behavior before those tasks land, implementers can defer PR-only behavior, size limits, or review gating; the failure is a policy gap during staged rollout.

### 8.2 Rebuttals / revisions (`software-engineer`)

- F-1 addressed: `docs/workflow-pipeline.md` defines one M7 run as exactly one selected framework issue, issue URL, or framework-gap identifier, with at most one branch, one proposal/update set, one bounded patch, and one PR.
- F-2 addressed: failure output must include target, failed stage, measured patch totals when available, and next-action hint in the workflow summary; issue comments are permitted only when issue write permission is available.
- F-3 addressed: PR-only behavior, pre-PR checks, patch limits, no-op/failure output, and review gating are binding in `docs/workflow-pipeline.md` now rather than deferred to later workflow tasks.
