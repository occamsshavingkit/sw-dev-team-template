## 2. Chosen ADR path

## 3. Implementation sketch

- **Module / file set touched:** `docs/framework-project-boundary.md` only.
- **Public interfaces introduced or modified:** none; this clarifies artifact authority vocabulary for existing framework/project boundary handling.
- **Key internal data shapes:** three authority classes: `canonical`, `generated`, and `ephemeral`.
- **Control flow in guidance:** add a short Documentation Authority Policy near the existing path ownership model; define each class, state that generated content must name canonical inputs and regeneration checks, and state that ephemeral artifacts are not durable sources of truth.
- **Manual mirror rule:** prohibit manual mirrored copies of canonical content; duplicated content must be generated, linked, or removed.
- **Non-goal:** do not reclassify every repository path in this task; keep path-specific rules where they already live.

## 4. Dependencies touched

- **Libraries added or bumped:** none.
- **Internal modules consumed:** existing `docs/framework-project-boundary.md` path ownership and commit/PR split guidance.
- **External services / APIs:** none.

## 5. Test plan outline

- **Documentation check:** verify `docs/framework-project-boundary.md` defines `canonical`, `generated`, and `ephemeral` in one policy section.
- **Prior-art coverage:** verify the section implements the recommendation in `docs/prior-art/T024.md` by prohibiting manual mirrors.
- **Boundary consistency:** verify the new terms do not contradict the existing framework-managed, project-filled, and project-owned layers.
- **Regression command:** run `git diff --check`.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Authority taxonomy becomes too broad to apply during review. | Keep definitions short and tie them to existing path classification work. |
| Generated artifacts are treated as canonical by implication. | Require generated artifacts to cite canonical inputs and regeneration checks. |
| Manual mirrors persist under a new label. | State that mirrored content must be generated, linked, or removed. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **Major:** If the policy defines `canonical`, `generated`, and
  `ephemeral` without an explicit decision path for ambiguous files,
  reviewers can classify the same artifact differently and still merge
  manual mirrors under different labels.
- **Major:** If generated artifacts are told to cite canonical inputs but
  not a concrete regeneration check, stale generated output can be treated
  as current during downstream upgrade or release review.
- **Minor:** If ephemeral artifacts are only described as non-durable but
  not excluded from decision records, agents may continue citing work logs
  or proposal scratch material as authoritative evidence after the task
  context has expired.

### 8.2 Rebuttals / revisions (`software-engineer`)

- Addressed: `docs/framework-project-boundary.md` now requires ambiguous
  artifacts to be classified against manifest/customization evidence and
  this boundary document before editing.
- Addressed: generated artifacts must identify canonical inputs and the
  regeneration or freshness check used before they are treated as current.
- Addressed: ephemeral artifacts are explicitly excluded from durable
  decision records and downstream authority.
