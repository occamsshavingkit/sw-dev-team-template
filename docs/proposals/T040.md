## 3. Implementation sketch

- **Module / file set touched:** create `scripts/compile-runtime-agents.sh`; write only generated runtime outputs under `docs/runtime/agents/` and `docs/runtime/agents/generated-artifacts.manifest.json`.
- **Public interfaces introduced or modified:** new repo-root command that deterministically compiles canonical agent inputs into runtime candidates, reports line counts unconditionally, and supports `LLMDC_BIN` / `SCHEMA2LLMD_BIN` overrides for LLMD tooling.
- **Key internal data shapes:** ordered canonical input list, per-role compiled Markdown output, generated-artifact manifest records in `docs/runtime/agents/generated-artifacts.manifest.json` compatible with `schemas/generated-artifact.schema.json`, and summary rows with path, line count, byte count, checksum, optional token count, and LLMD tool status.
- **Control flow:** verify repo root, resolve `/usr/local/bin/llmdc` and `/usr/local/bin/schema2llmd` unless overridden, enumerate `CLAUDE.md`, `AGENTS.md`, `.claude/agents/*.md`, and matching `*-local.md` supplements in stable order, render generated candidate files with provenance headers, run `schema2llmd` against `schemas/generated-artifact.schema.json`, run `llmdc` against each generated runtime candidate, compute checksums and line/byte counts for retained generated Markdown outputs, and print a deterministic summary.
- **Token / LLMD strategy:** line counts are mandatory; token counts remain `not-configured` because the LLMD tools do not provide provider token counts. `llmdc` and `schema2llmd` are used as compiler/conversion evidence when available, and absence or unusable overrides is reported instead of inventing LLMD or token evidence.
- **Authority rule:** generated outputs must say they are subordinate to canonical inputs and must not introduce a parallel role roster, escalation chain, or customer interface.

## 4. Dependencies touched

- **Libraries added or bumped:** none. Existing LLMD tools are used from `/usr/local/bin` by default, with `LLMDC_BIN` and `SCHEMA2LLMD_BIN` overrides.
- **Internal modules consumed:** `CLAUDE.md`, `AGENTS.md`, `.claude/agents/*.md`, `.claude/agents/*-local.md` if present, `docs/runtime/agents/README.md`, `schemas/generated-artifact.schema.json`, `scripts/lib/manifest.sh`, and `docs/prior-art/T040.md`.
- **External services / APIs:** none required. Local tools used when present: `/usr/local/bin/llmdc` and `/usr/local/bin/schema2llmd`.

## 5. Test plan outline

- **Syntax:** run `bash -n scripts/compile-runtime-agents.sh`.
- **Determinism:** run the compiler twice and confirm generated files and manifest checksums are stable.
- **Authority check:** verify generated outputs cite canonical inputs and state non-canonical/generated status.
- **Local supplement check:** add a temporary `*-local.md` fixture in a disposable copy and confirm it is included or reported in provenance.
- **Reporting check:** confirm every generated role reports line and byte counts, with token counts clearly marked as unavailable when no provider counter is configured, and confirm the manifest records LLMD tool usage or absence in generator metadata.
- **Regression command:** run `git diff --check` and T039 lint after generated manifests exist.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Generated files drift from canonical inputs. | Include canonical input list, generator metadata, checksums, and deterministic ordering. |
| Token counts are misread as provider-accurate. | Make line/byte counts mandatory, keep token counts `not-configured`, and record LLMD compiler usage separately from provider token counting. |
| Compiler creates a parallel role model. | Render only thin runtime candidates subordinate to canonical role files. |
| Generated outputs overwrite hand edits. | Treat generated directory as non-canonical and use checksums/freshness checks to expose manual drift. |
| Manifest placement is ambiguous and validators miss generated-runtime provenance. | Emit exactly `docs/runtime/agents/generated-artifacts.manifest.json` next to runtime candidates and validate it through T039/G6. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1:** Compiler overwrites reviewed runtime candidates without drift guard — **How it fails:** rerunning the script replaces files under `docs/runtime/agents/` even when prior hand-review annotations or manual changes exist, destroying evidence before review. — **Severity if unaddressed:** blocker.
- **F-2:** Local supplement ordering is nondeterministic — **How it fails:** filesystem-dependent enumeration changes compiled prompt order or provenance, so repeated runs produce different checksums from identical inputs. — **Severity if unaddressed:** major.
- **F-3:** Token fallback is ambiguous in summary output — **How it fails:** missing provider token tooling is reported as `0`, blank, or estimated without labeling, causing reviewers to treat line/byte evidence as provider-accurate token evidence. — **Severity if unaddressed:** major.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 addressed:** `scripts/compile-runtime-agents.sh` writes M6 output under `docs/runtime/agents/generated/`, leaving existing reviewed M0/M1 candidates untouched; only files inside that generated subdirectory are replaced by the generator.
- **F-2 addressed:** canonical role files and local supplements are enumerated in sorted order, and each role's inputs are built deterministically from `CLAUDE.md`, `AGENTS.md`, the role file, and a matching supplement when present.
- **F-3 addressed:** summary and manifest output always label token evidence as `not-configured` unless a future provider-specific counter is explicitly added; line and byte counts are mandatory.

### 8.3 Status

- Round: 1
- Outcome: closed with generated-subdirectory guard, stable ordering, and explicit token fallback
