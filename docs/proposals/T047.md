## 8. Duel

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** If the workflow treats the warning-only question lint as a required failure gate without confirming the script contract, benign warning output can block unrelated PRs; the failure is a governance visibility check becoming an accidental merge stopper.
- **F-2 [Major]:** If coverage excludes `AGENTS.md`, local agent supplements, or docs that define escalation/question behavior, new multi-question prompts can bypass lint visibility; the failure is customer-inbox flooding despite a green question-lint workflow.
- **F-3 [Critical]:** If inline shell parses changed file contents or PR metadata instead of invoking the local script directly with quoted paths, crafted markdown or filenames can alter command execution; the failure is a read-only lint job becoming an injection surface.

### 8.2 Rebuttals / revisions (`software-engineer`)

- F-1 addressed: `.github/workflows/question-lint.yml` invokes the existing warning-only script directly and does not add fatal parsing around warning text.
- F-2 addressed: the workflow runs on all PRs and manual dispatch without narrow path filters, covering `AGENTS.md`, agent files, docs, and lint-script changes.
- F-3 addressed: the workflow invokes `scripts/lint-questions.sh` directly and does not parse PR metadata or changed-file contents inline.
