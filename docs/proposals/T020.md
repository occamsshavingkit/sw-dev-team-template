## 3. Implementation sketch

- **Module / file set touched:** add `scripts/lint-questions.sh`; no other production files unless a test harness or docs index reference is explicitly requested in review.
- **Public interfaces introduced or modified:** new executable command `scripts/lint-questions.sh` run from repository root; optional positional path arguments may be avoided for the first cut to keep the contract minimal.
- **Key internal data shapes:** a fixed list of candidate files and simple warning records shaped as `WARN: <path>:<line>: <rule>: <message>`.
- **Control flow:** verify repo-root context, scan only existing target files, run lightweight text heuristics for known bad patterns, print all warnings, and exit `0` for warnings; exit non-zero only for usage/runtime errors.
- **Initial scan surfaces:** `docs/FIRST_ACTIONS.md`, `docs/templates/scoping-questions-template.md`, `.claude/agents/tech-lead.md`, `docs/OPEN_QUESTIONS.md`, and `docs/intake-log.md` when present, expanding `docs/prior-art/T020.md` coverage to include the active scaffold seed source while excluding the scoping template's advisory follow-up catalog from bundled-prompt block checks.
- **Rules proposed:** flag likely compound customer-facing questions, multiple numbered customer-question items in one prompt block, multiple independent option sets in one customer prompt, non-empty `agents-running-at-ask` values, and compound-looking `docs/OPEN_QUESTIONS.md` rows.
- **Non-goal:** do not hard-fail CI for content warnings in this task.

## 5. Test plan outline

- **Unit / script behavior:** run `scripts/lint-questions.sh` against the current repository and confirm it exits `0` while printing deterministic `WARN:` lines for any matched patterns.
- **Positive fixtures:** create temporary fixture copies under `/tmp` or a disposable test directory containing one known-bad compound prompt, one non-empty `agents-running-at-ask`, and one multi-question `OPEN_QUESTIONS.md` row; confirm each warning appears.
- **Negative fixtures:** verify one atomic question with a single option set does not warn as compound solely because it contains valid options.
- **Missing-file behavior:** temporarily run from a fixture missing `docs/intake-log.md` and confirm lint skips absent optional files without failing.
- **Regression command:** run `bash -n scripts/lint-questions.sh` and `git diff --check`.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| False positives make the warning stream noisy and ignored. | Keep warnings specific, include rule names, and avoid hard failure in this release cycle. |
| Regex heuristics miss real compound prompts. | Target known historical bad patterns from `docs/prior-art/T020.md` first; defer stricter semantic linting to a later task. |
| Script behaves differently across shells. | Use Bash explicitly via shebang and validate with `bash -n`. |
| Lint accidentally fails CI on content warnings. | Exit `0` after warnings; reserve non-zero exit for invocation or runtime errors. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** If `scripts/lint-questions.sh` scans only the initial fixed file list and silently skips missing `docs/intake-log.md` or future customer-question surfaces, production users can keep asking compound customer prompts outside the covered files while the lint reports clean; the implementation needs a visible skip warning or documented surface list tied to `docs/prior-art/T020.md` acceptance coverage.
- **F-2 [Major]:** If the compound-question regex treats every option list or every `?` sequence as independent questions, valid atomic prompts with one decision axis and several options will generate noisy warnings; once users learn the warning stream is mostly false positives, the warning-only control stops preventing the exact behavior it is meant to catch.
- **F-3 [Critical]:** If any warning path exits non-zero through `set -e`, failed `grep`, missing optional files, or a pipeline status, CI or local automation can start hard-failing before the planned warning-only release cycle, blocking unrelated scaffold work despite the proposal's explicit non-goal.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 revision:** The script reports every configured optional surface that is absent as `WARN: <path>: optional scan surface missing`, scans the canonical surfaces from `docs/prior-art/T020.md`, and now also treats `docs/templates/scoping-questions-template.md` as a required scan surface because it is the active scaffold seed source.
- **F-2 revision:** The checks do not treat `options-presented:`, a single option set, one wrapped numbered/bulleted question item, or the scoping template's advisory follow-up catalog as a violation; warnings target multiple question marks, repeated question-list items in one customer-facing prompt block, non-empty `agents-running-at-ask`, and compound-looking `OPEN_QUESTIONS.md` rows.
- **F-3 revision:** Content findings only increment a warning counter and the script prints a warning-only summary before returning `0`; only a non-repository invocation returns non-zero.
