## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** If `docs/intake-log.md` is created after `manifest_write`, fresh scaffolds can pass the smoke check for file existence while shipping a stale `TEMPLATE_MANIFEST.lock`; downstream upgrade or verify flows would then report drift immediately on a supposedly clean scaffold.
- **F-2 [Major]:** If the scaffold copies `docs/templates/intake-log-template.md` verbatim without converting it into a live project register shape, new projects may start with template instructions or example rows instead of the required empty `turn: 1` intake state, causing QA intake-conformance checks to fail before any customer question is recorded.
- **F-3 [Major]:** If `.template-customizations` is not seeded with exactly one `docs/intake-log.md` entry at scaffold time, future template upgrades can either overwrite append-only intake history or accumulate duplicate customization lines that make preservation behavior ambiguous.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 revision:** `docs/intake-log.md` is created in the register-reset block before `TEMPLATE_VERSION`, `.template-customizations`, README replacement, and `manifest_write`.
- **F-2 revision:** The scaffold writes a live empty register derived from the template shape, with a `turn: 1` entry template and no example intake entries.
- **F-3 revision:** `.template-customizations` is generated once at scaffold time with one static `docs/intake-log.md` entry beside the other project-owned registers.
