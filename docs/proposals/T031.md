## 2. Chosen ADR path

## 3. Implementation sketch

- **Module / file set touched:** `docs/model-routing-guidelines.md` only.
- **Public interfaces introduced or modified:** no code API; document provider/model ID notation and Gemini model classes as routing-policy mappings.
- **Key internal data shapes:** provider-neutral tiers, provider/model ID convention, OpenCode provider notation, Gemini Pro/Flash class mappings, release-time exact-ID verification evidence.
- **Control flow in guidance:** add a provider/model ID convention section near the current policy shape; extend the tier mapping table or adjacent provider notes to include Gemini class guidance without hard-pinning exact IDs.
- **Verification rule:** exact Gemini and OpenCode provider model IDs remain advisory until verified before tagged release, consistent with the existing release-time model-ID rule.
- **Non-goal:** do not change role authority, tier defaults, fallback behavior, or logging format; T032 and T033 cover fallback rules.

## 4. Dependencies touched

- **Libraries added or bumped:** none.
- **Internal modules consumed:** existing `docs/model-routing-guidelines.md` tier, effort, Codex effort, plan-mode, role defaults, and governance sections.
- **External services / APIs:** Gemini and OpenCode provider/model naming are documented as classes/conventions only; no unverified exact IDs are introduced.

## 5. Test plan outline

- **Documentation check:** verify provider/model ID convention is explicit and does not make one provider mandatory.
- **Mapping check:** verify Gemini Pro/Flash classes map to existing tiers or task classes without changing role authority.
- **Release check:** verify exact Gemini/OpenCode IDs are subject to release-time verification before tagged release.
- **Prior-art coverage:** verify the implementation follows `docs/prior-art/T031.md` and rejects stale exact-ID pinning.
- **Regression command:** run `git diff --check`.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Exact Gemini IDs are guessed and become stale. | Use model classes in this task and require release-time exact-ID verification. |
| Provider syntax is mistaken for mandatory downstream tooling. | Document notation as a convention under provider-neutral tiers, not a vendor requirement. |

| T032/T033 fallback scope leaks into this task. | Limit this proposal to provider/model notation and model classes. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** The provider/model notation can fail in production if examples look like valid exact OpenCode or Gemini IDs; downstream users may copy stale values as release facts despite the proposal saying exact IDs require release-time verification.
- **F-2 [Major]:** Gemini Pro/Flash mappings can be misread as new role defaults if they are placed in the main tier table without explicit provider-advisory labeling, causing QA, research, PM, or writing roles to appear governed by vendor classes instead of the existing role and effort policy.
- **F-3 [Minor]:** The non-goal excludes fallback behavior, but the same edited file will also receive T032/T033 fallback changes; without clear section boundaries, the implementation can accidentally mix provider notation with fallback authority rules and make later audit of T031 scope ambiguous.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 closed:** Provider examples use `*-class` labels and the provider/model convention states exact IDs require release-time verification before pinning.
- **F-2 closed:** Gemini mappings are in advisory provider-class sections under the existing tier policy and explicitly do not change role authority or required review.
- **F-3 closed:** Provider/model notation and fallback behavior now live in separate sections, making T031 scope auditable.
