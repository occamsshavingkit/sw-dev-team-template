## 3. Implementation sketch

- **Public interfaces introduced or modified:** no code API; add OpenCode adapter guidance for Codex/OpenCode-compatible harness use.
- **Key internal data shapes:** generated or generator-backed adapter, canonical inputs, local supplement loading, no full-role-text duplication, freshness/lint expectation, role output format preservation.

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** Guidance can still fail in production if it permits "generator-backed" adapters without naming the generator inputs and freshness signal; maintainers may hand-edit thin wrappers and treat them as current even after `.claude/agents/*.md` or local supplements change.
- **F-2 [Major]:** The adapter can require reading canonical role files but not require preserving the role's output format, causing OpenCode wrappers to return harness-native summaries that break downstream QA, code-review, or escalation workflows.
- **F-3 [Major]:** Because the write target is `AGENTS.md`, the guidance can accidentally make OpenCode rules appear Codex-only or top-level-only; if specialist adapters do not explicitly read matching `*-local.md` supplements, project-specific QA/security constraints can be bypassed at runtime.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 closed:** `AGENTS.md` now requires generated or generator-backed adapters to name canonical inputs and pass a regeneration, lint, or freshness check before operational use.
- **F-2 closed:** `AGENTS.md` now requires OpenCode wrappers to preserve the role's required output format and review path.
- **F-3 closed:** `AGENTS.md` frames OpenCode as a harness/provider adapter, not Codex-only guidance, and requires matching local supplement loading.
