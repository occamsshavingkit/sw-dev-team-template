## 1. Task reference

- **Task ID:** T033

- **Trigger clauses that fire:** 3 — cross-module boundary.

- **ADR reference:** none.

## 2. Chosen ADR path

## 3. Implementation sketch

- **Module / file set touched:** `docs/model-routing-guidelines.md` only.
- **Public interfaces introduced or modified:** no code API; define a required fallback evidence record for model/provider substitutions.
- **Key internal data shape:** a compact YAML-like record with `task_id`, `agent`, `requested_model`, `actual_model`, `fallback_reason`, `timestamp`, and an authority-preservation statement.
- **Control flow in guidance:** add the logging format adjacent to fallback behavior; require records when the executed provider/model differs from the requested routing decision.
- **Authority rule:** fallback records are evidence only; changing model provider does not change role ownership, canonical source hierarchy, output format, review obligations, or customer-interface rules.
- **Safety rule:** fallback reasons should be categorical and sanitized, not copied provider error payloads.

## 4. Dependencies touched

- **Libraries added or bumped:** none.
- **Internal modules consumed:** existing `docs/model-routing-guidelines.md` governance and effort policy, `AGENTS.md` pre-close evidence expectations, and `docs/prior-art/T033.md`.
- **External services / APIs:** none; the log format is provider-neutral and intentionally avoids raw provider error payloads.

## 5. Test plan outline

- **Format check:** verify fallback log fields include agent, requested model, actual model, fallback reason, timestamp, and task ID.
- **Authority check:** verify the rule preserves role ownership, output format, review obligations, and canonical source hierarchy.
- **Sanitization check:** verify fallback reasons are categorical and do not invite secret-bearing raw error copies.
- **Prior-art coverage:** verify wording implements `docs/prior-art/T033.md` without requiring full transcripts.
- **Regression command:** run `git diff --check`.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Fallback evidence cannot be audited later. | Require timestamp, task ID, requested model, actual model, and reason fields. |
| Log records leak provider errors or secrets. | Use categorical fallback reasons and prohibit raw provider payload copying. |
| Fallback records become a new authority source. | State records are evidence only and preserve original role/source hierarchy. |
| Logging format is too heavy for routine use. | Keep the record compact and avoid transcript storage. |

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **F-1 [Major]:** The fallback record omits `requested_tier` and `requested_effort`, so an audit can know the requested and actual models but still miss that a frontier / `xhigh` routing decision was downgraded to a lower-risk execution class.
- **F-2 [Major]:** A categorical `fallback_reason` alone can be too vague for release review; if values are not enumerated or constrained, teams may enter free-text explanations that either leak provider details or hide whether the fallback was caused by policy, outage, quota, or local configuration.
- **F-3 [Minor]:** The authority-preservation statement can become boilerplate that is copied even when the executed output format changed; the format should require a concrete assertion that role output format, review gate, and canonical input hierarchy were preserved for that fallback event.

### 8.2 Rebuttals / revisions (`software-engineer`)

- **F-1 closed:** The fallback record now includes `requested_tier` and `requested_effort`.
- **F-2 closed:** `fallback_reason` values are enumerated and raw provider error payloads are prohibited.
- **F-3 closed:** The record requires a concrete `authority_preserved` assertion covering role output format, review gate, and canonical inputs.
