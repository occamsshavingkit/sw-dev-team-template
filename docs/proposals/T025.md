## 3. Implementation sketch

- **Module / file set touched:** `docs/framework-project-boundary.md` only.
- **Public interfaces introduced or modified:** none; this documents existing roadmap ownership semantics.
- **Key internal data shapes:** root `ROADMAP.md` as upstream template release-planning artifact; downstream product roadmaps as project-owned only when intentionally created or retained.
- **Control flow in guidance:** add root roadmap classification to the release/version or path-ownership guidance; state that template root `ROADMAP.md` is excluded from downstream scaffold and upgrade ship files.

## 8. Duel

**Solution Duel** — pre-code adversarial review. `qa-engineer` writes
three concrete ways this proposal could fail before implementation.

### 8.1 Findings (`qa-engineer`)

- **Critical:** If the guidance says root `ROADMAP.md` is template-only
  without the classification-before-editing rule, a downstream maintainer
  can delete a legitimate product roadmap as supposed template leakage.
- **Major:** If the documentation states exclusion intent but does not
  reference current scaffold, manifest, and upgrade behavior, future
  maintainers may "fix" scripts that are already correct and reintroduce
  root roadmap shipping.
- **Major:** If framework-repair evidence is not tied to commit/PR split
  guidance, roadmap cleanup can be mixed into product commits and make it
  impossible to audit whether product roadmap changes were intentional.

### 8.2 Rebuttals / revisions (`software-engineer`)

- Addressed: `docs/framework-project-boundary.md` requires classification
  before editing downstream root `ROADMAP.md` and preserves intentional
  project-owned roadmaps.
- Addressed: the guidance states current scaffold and upgrade behavior:
  template root `ROADMAP.md` is excluded from downstream shipping.
- Addressed: template roadmap leakage is classified as framework-repair
  evidence, not ordinary product roadmap work, preserving commit/PR split.
