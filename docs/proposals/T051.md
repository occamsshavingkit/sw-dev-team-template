# Proposal — T051

## 1. Task reference

- **Task ID:** T051
- **One-line task statement:** Enforce human review for every M7 automation PR before merge and document the required reviewer role sequence in `docs/workflow-pipeline.md`.
- **Trigger clauses that fire:** 3 — cross-module boundary.
- **Prior-art reference:** `docs/prior-art/T051.md`.
- **ADR reference:** none.

## 2. Chosen ADR path

- **Alternative chosen:** none — no ADR.
- **Reason:** This task documents and enforces the existing human review rule for M7 PRs; no ADR exists.

## 3. Implementation sketch

- Touch `docs/workflow-pipeline.md`; optionally align `.github/workflows/improve-template.yml` if it needs reviewer metadata or PR body text.
- Document that every M7 automation PR requires human review before merge.
- Required role sequence before merge:
  1. Implementation owner verifies the PR matches the selected issue/gap, stays within T050 size limits, and contains no unrelated changes.
  2. `release-engineer` reviews workflow/release mechanics, branch/PR behavior, checks, and rollback/no-op behavior.
  3. `security-engineer` reviews every M7 automation PR that touches workflows, permissions, tokens, generated bot behavior, auth/authz, secrets, PII, or network-exposed surfaces; for docs-only or policy-only automation PRs with none of those surfaces, record `security-engineer: not required` in the PR review checklist.
  4. `code-reviewer` performs final review and approval per `CLAUDE.md` Hard Rule #3 after the preceding role checks are complete.
- State automation must not approve or merge its own PR.
- State any new commit after a role's approval invalidates that role's approval and requires re-review from that role and then final `code-reviewer` review again.
- Note branch protection or repository rulesets are the enforcement surface where available; committed docs define the policy even when repo settings are configured separately.

## 4. Dependencies touched

- No package dependencies.
- Internal dependencies: `docs/workflow-pipeline.md`, `CLAUDE.md` Hard Rule #3, `.github/workflows/improve-template.yml` if PR body/reviewer hints are added.
- External service: GitHub pull request reviews and branch protection, per `docs/prior-art/T051.md`.

## 5. Test plan outline

- Unit: documentation lint and link/path review.
- Integration: confirm `improve-template.yml` cannot approve or merge and that PR body/check output names required reviewers if implemented there.
- System / acceptance: manual review that M7 PR policy satisfies FR-013 and FR-014 and includes `code-reviewer` approval.
- Performance / security: confirm no workflow permission grants enable self-approval or bypass of branch protection.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Docs promise enforcement that repo settings do not provide. | Separate policy from branch-protection configuration and identify settings as repository-admin enforcement. |
| Reviewer sequence blocks harmless docs-only automation. | Require an explicit `security-engineer: not required` checklist entry when no security-sensitive surface is touched. |
| Automation pushes after approval. | Document that post-approval commits require fresh review and prohibit self-merge. |

## 7. Open questions

- None for code start. Decision: `security-engineer` review is required for security-sensitive M7 automation PRs; otherwise the PR checklist must explicitly record `security-engineer: not required` before final `code-reviewer` review.

## 8. Duel

### 8.1 Findings (`qa-engineer`)

- **F-1 [Critical]:** If the policy records reviewer roles only in prose and the PR body/checklist does not require explicit role attestations, an M7 automation PR can merge with generic human approval but without the release/security/code-review sequence; the failure is apparent compliance with no auditable review chain.
- **F-2 [Major]:** If new commits after role approval do not clearly invalidate the affected approval and final `code-reviewer` review, automation can push a changed diff under stale approvals; the failure is reviewed content differing from merged content.
- **F-3 [Major]:** If `security-engineer: not required` is allowed without criteria tied to workflows, permissions, tokens, bot behavior, PII, or network-exposed surfaces, risky automation PRs can self-classify as docs-only; the failure is security review skipped on a permission-bearing change.

### 8.2 Rebuttals / revisions (`software-engineer`)

- F-1 addressed: `docs/workflow-pipeline.md` requires explicit PR-body or checklist entries for implementation owner, `release-engineer`, conditional `security-engineer`, and final `code-reviewer`; `improve-template.yml` includes that checklist in generated PR bodies.
- F-2 addressed: `docs/workflow-pipeline.md` states new commits after a role approval invalidate that role's approval and require re-review before final `code-reviewer` approval.
- F-3 addressed: the conditional security rule is tied to workflows, permissions, tokens, generated bot behavior, auth/authz, secrets, PII, and network-exposed surfaces; otherwise the checklist must record `security-engineer: not required` with criteria.
