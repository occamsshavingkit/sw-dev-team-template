## A. Problem Statement and Root Cause

### A.1 Observed violation

A `librarian` subagent dispatched to append a verbatim customer answer to
`CUSTOMER_NOTES.md` surfaced an interactive confirmation prompt to the
human operator (the customer). This is a Hard Rule #1 violation: only
`tech-lead` may interface with the customer; subagents must not.

### A.2 Layer-1 root cause — `customer-notes-guard.py` is role-blind

`scripts/hooks/customer-notes-guard.py` is a Claude Code `PreToolUse` hook
that fires on every Write / Edit / MultiEdit / Bash event touching
`CUSTOMER_NOTES.md`. Its `_approval_gate_output()` function (lines 531–551)
returns `permissionDecision: "ask"` unconditionally:

```python
# scripts/hooks/customer-notes-guard.py  ≈ line 548 (current state)
return {
    "hookSpecificOutput": {
        "hookEventName": "PreToolUse",
        "permissionDecision": "ask",
        "permissionDecisionReason": reason,
    }
}
```

The header comment at line 7 records the design assumption:

> "The hook input does not provide a stable role identity, so this is an
> approval gate rather than a role detector."

That assumption was correct when the hook was written but has been
superseded: `tech-lead-authoring-guard.py` (its "sister hook" per the
comments, kept "verbatim in sync") demonstrates at lines 808–870 that
`agent_type` / `subagent_role` fields in the hook event payload reliably
carry the spawned specialist's role identity in v1.5.2 and later. The
customer-notes guard simply has not been updated to use them.

`main()` in the customer-notes guard (lines 554–591) follows this path for
every `CUSTOMER_NOTES.md` write, with no role check:

```
_tool_input_touches_customer_notes() → True
_extract_candidate_content() / _content_findings()   [advisory only]
print(json.dumps(_approval_gate_output(findings)))   # always "ask"
```

A hook decision of `"ask"` is an **explicit** harness instruction. It
overrides any project-level `defaultMode: acceptEdits` that a downstream
project might set. Even in a well-configured downstream project, the
librarian's `CUSTOMER_NOTES.md` write therefore always produces a human
prompt — routed to the terminal that spawned the main session = the
customer.

### A.3 Layer-2 root cause — scaffold ships no `permissions` block

`sw-dev-team-template/.claude/settings.json` contains no `permissions`
block and no `defaultMode` key. The role-gated hooks (`tech-lead-authoring-
guard.py`) return `None` (Python `None` → no stdout output) for authorized
writes, which the harness interprets as "no hook opinion". In the absence of
an explicit `defaultMode: "acceptEdits"`, the harness falls back to its own
interactive default for spawned subagents, which is effectively `"ask"` —
routed to the team leader = the human terminal.

The two layers compound: even if Layer-1 were fixed so that the customer-
notes guard emits explicit `allow` for `librarian`, a separate authorized
write by another hook (or a future hook that uses silent passthrough for its
allow cases) would still fall through to the interactive default. The scaffold
needs a defensive floor.

### A.4 Why the fix belongs in v1.5.3

v1.5.2 introduced the reliable `agent_type` read in `tech-lead-authoring-
guard.py`. The evidence that it works on subagent Write/Edit events is that
the librarian write passed the authoring guard (which uses `agent_type`)
and was only stopped by the customer-notes guard (which does not). Fixing
the customer-notes guard is a surgical patch on top of the v1.5.2 baseline.
No deep architectural change is needed.

## B. Layer-1 Design: Role-Aware `customer-notes-guard.py`

### B.1 Shared role-resolution helper

The role-resolution logic in `tech-lead-authoring-guard.py`
(`_event_subagent_role`, `_active_dispatch_subagent_role`,
`_resolve_subagent_role`, lines 808–870) is non-trivial:

- reads `agent_type` and `subagent_role` (compatibility alias) from the
  hook event dict;
- falls back to `docs/pm/active-dispatches.json` via
  `ANTIGRAVITY_CONVERSATION_ID` for Antigravity harnesses;
- validates the resolved value through `_validate_role()` (rejects
  `"tech-lead"` self-push; accepts `CANONICAL_ROLES` members and
  `sme-*` slugs).

Duplicating this logic verbatim into the customer-notes guard would
immediately violate the "verbatim in sync" contract the two hooks already
struggle to maintain. Instead:

**Create `scripts/hooks/lib/role_resolver.py`** as a new shared module
alongside the existing `scripts/hooks/lib/roles.py`. It exports one public
function:

```python
def resolve_subagent_role(event: dict) -> tuple[str | None, str | None]:
    """Return (role, source) for a positively-identified specialist.

    Returns (None, None) when the caller cannot be identified as a
    recognized specialist (main-session / harness default / unknown payload).
    Never returns "tech-lead" as a role — that is filtered by
    _validate_role() in the underlying implementation.
    """
```

The implementation is the body of `_resolve_subagent_role` plus its
callees (`_event_subagent_role`, `_active_dispatch_subagent_role`,
`_validate_role` delegate) lifted directly from the authoring guard, with
the `CANONICAL_ROLES` / `SME_ROLE_RE` imports satisfied from
`scripts/hooks/lib/roles.py`. The authoring guard's private implementations
become thin wrappers that delegate to the shared function — no behavior
change in the authoring guard, but the logic now lives in one place.

`customer-notes-guard.py` adds a single import:

```python
import sys, os
sys.path.insert(0, os.path.join(os.path.dirname(__file__), ".."))
from lib.role_resolver import resolve_subagent_role  # noqa: E402
```

(Pattern already established by `tech-lead-authoring-guard.py` line 71 for
`roles.py`; the `sys.path` insertion keeps both hooks executable as
standalone scripts and importable in test harnesses.)

### B.2 Decision matrix

After resolving the caller role, `customer-notes-guard.py` applies the
following deterministic matrix:

| Resolved role | Source | Decision | Rationale |
|---|---|---|---|
| `"librarian"` | `agent_type` / `subagent_role` / Antigravity fallback | **`allow`** + stderr audit log | Librarian is the sole authorized `CUSTOMER_NOTES.md` steward per the template contract. An explicit `allow` prevents the harness from escalating to interactive ask. |
| Any other recognized specialist (e.g. `"software-engineer"`, `"architect"`, `"sme-*"`) | same sources | **`deny`** | These roles have no `CUSTOMER_NOTES.md` write authorization. The deny reason instructs the specialist to route through tech-lead → librarian. |
| `None` / `"tech-lead"` / unrecognized string | no source or payload absent | **`ask`** | This is the genuine-human-edit checkpoint. When `agent_type` is absent, the caller is the main session (= tech-lead = the human terminal). Human-initiated edits to customer truth are the legitimate use case for a confirmation gate. Unrecognized strings (bogus role values) are treated as unknown and conservatively kept at `ask` rather than `deny` so that future harness variants do not silently block legitimate human edits. |

**Rationale for `ask` vs `deny` for main-session/unknown:** Converting
unknown callers to `deny` would break the only legitimate human-edit path
(direct tech-lead/human maintenance of `CUSTOMER_NOTES.md`). The trade-off
is that a subagent sending a payload without `agent_type` also gets `ask`
rather than `deny`. This is the safe-side failure: it produces a prompt
(undesirable but not a Hard Rule violation) rather than silently allowing an
unauthorized write. If evidence accumulates that all harness versions
populate `agent_type` on spawned subagents, the unknown branch can be
tightened to `deny` in a future patch without changing the `librarian` →
`allow` path.

**`tech-lead` specific:** `_validate_role()` in the shared resolver already
rejects `"tech-lead"` as a return value (same rule as the authoring guard:
tech-lead cannot use its own identity to bypass a guard). A hook event
carrying `agent_type: "tech-lead"` therefore resolves to `(None, None)` →
falls into the `ask` branch. This is correct: a main-session write where the
harness somehow populates `agent_type: "tech-lead"` is indistinguishable
from a human edit and should get the human checkpoint.

### B.3 New output helpers in `customer-notes-guard.py`

Add two new output functions alongside the existing `_approval_gate_output`:

```python
def _allow_output(role: str, source: str) -> dict:
    """Emit explicit allow for a positively-identified authorized steward."""
    return {
        "hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "allow",
            "permissionDecisionReason": (
                f"customer-notes-guard: {source}={role!r} is the authorized "
                "CUSTOMER_NOTES.md steward."
            ),
        }
    }


def _deny_unauthorized_output(role: str, source: str) -> dict:
    """Deny write by a recognized but unauthorized specialist."""
    return {
        "hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "deny",
            "permissionDecisionReason": (
                f"FW-HR1: {source}={role!r} may not write CUSTOMER_NOTES.md directly. "
                "Route customer answers through tech-lead → librarian. "
                "See template contract: CLAUDE.md § 'Hard rules' #1."
            ),
        }
    }
```

The existing `_approval_gate_output()` (returning `"ask"`) is **retained
unchanged** and becomes the main-session / unknown branch.

### B.4 Updated `main()` flow

```
event = json.load(stdin)
if not _tool_input_touches_customer_notes(tool_input): return 0   # unchanged

role, source = resolve_subagent_role(event)          # NEW: shared helper

if role == "librarian":
    findings = _content_findings(...)                # advisory, still runs
    sys.stderr.write(audit-log line)                 # audit trail
    print(json.dumps(_allow_output("librarian", source)))
    return 0

if role is not None:   # recognized specialist, not librarian
    print(json.dumps(_deny_unauthorized_output(role, source)))
    return 0

# role is None: main-session, tech-lead rejected by validator, or unknown
findings = _content_findings(...)                    # advisory, still runs
print(json.dumps(_approval_gate_output(findings or None)))
return 0
```

Content findings continue to run for the `librarian → allow` path and are
emitted to `stderr` (not as a blocking gate) so the audit trail captures
any structural anomalies in the entry. They are not suppressed just because
the role is authorized.

## C. Layer-2 Design: `settings.json` Permissions Block and Explicit Hook Allows

### C.1 Add `defaultMode: "acceptEdits"` to scaffold `settings.json`

The root cause of the Layer-2 leak is that no `defaultMode` is set. Silent
hook passthrough (`None` / no stdout) therefore falls through to the
harness interactive default. The fix is minimal:

```jsonc
// sw-dev-team-template/.claude/settings.json  — diff
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
+ "permissions": {
+   "defaultMode": "acceptEdits"
+ },
  "env": { ... },
  "hooks": { ... }
}
```

`defaultMode: "acceptEdits"` means: when no hook fires, or when a hook
produces no stdout (silent passthrough = the current authorized-write path
in `tech-lead-authoring-guard.py`), the harness proceeds without prompting.
This is the correct floor for a project where role-gated hooks are the
primary access-control mechanism.

**Why not `permissions.allow` path patterns?** Static `allow` entries in
settings.json are not role-scoped — they bypass hooks for all callers.
Adding `Write(CUSTOMER_NOTES.md)` to `permissions.allow` would allow any
subagent to write it unconditionally, defeating the gate entirely. The hook
is the correct place for role-scoped decisions; `defaultMode` sets the floor
for the uncontested cases.

**Per-specialist tool scoping:** The `.claude/agents/*.md` front matter
`tools:` lines (e.g., `librarian.md`: `Read, Write, Edit, Grep, Glob,
SendMessage`) define the capability envelope per role. These are harness-
enforced at agent spawn time and are orthogonal to the `permissions` block
(which governs per-tool-invocation decisions). The hook layer gates specific
path writes *within* the already-constrained tool set. No additional
`permissions.allow` entries are needed to satisfy the per-specialist scoping
requirement — the existing `tools:` declarations already do that.

### C.2 Explicit `allow` from `tech-lead-authoring-guard.py` for authorized writes

Currently, the authoring guard returns Python `None` (no stdout) for
authorized writes — relying on the harness default. After adding
`defaultMode: "acceptEdits"`, this is safe, but it is fragile: a future
settings.json change that removes or changes `defaultMode` would silently
re-enable interactive prompts for all authorized specialist writes.

Introduce an `_allow_output()` helper in `tech-lead-authoring-guard.py`
(same signature as the one designed for customer-notes-guard in B.3) and
have `_decide_for_path()` and `_decide_for_command()` return it instead of
`(None, audit_callable)` for the authorized-role branch:

```python
# In _decide_for_path() — replace the silent-passthrough return:
# Before:  return None, lambda: _audit_override(...)
# After:
return _allow_output(_normalise(path), role, kind, source), \
       lambda: _audit_override(...)
```

The audit stderr log (`_audit_override`) still fires. The only behavioral
change is that the hook now emits an explicit JSON `allow` rather than
silence. The harness interprets both the same way in the presence of
`defaultMode: "acceptEdits"`, but explicit is more robust and more auditable.

**`main()` output path:** The existing `if decision is not None: print(...)` 
branch handles both `deny` and (new) `allow` outputs uniformly — no change
to the top-level dispatch.

## D. Test Plan

### D.1 Unit tests for role-aware `customer-notes-guard.py`

Add cases to `tests/hooks/test-customer-notes-guard.sh`. The existing test
harness already provides `_run_payload()` and `mkpayload*` helpers that feed
crafted JSON on stdin and assert on stdout. Add a new helper:

```bash
mkpayload_with_role() {
    # Build a Write tool_input payload carrying agent_type for a given role.
    python3 -c 'import json,sys
role,fp=sys.argv[1],sys.argv[2]
print(json.dumps({"agent_type":role,"tool_input":{"file_path":fp}}))
' "$1" "$2"
}
```

New cases (all against `CUSTOMER_NOTES.md` Write path):

| # | Scenario | `agent_type` value | Expected decision | Expected keyword in stdout |
|---|---|---|---|---|
| R-1 | Librarian write | `"librarian"` | allow | `"allow"` |
| R-2 | Software-engineer write | `"software-engineer"` | deny | `"deny"` |
| R-3 | Architect write | `"architect"` | deny | `"deny"` |
| R-4 | SME write | `"sme-domain"` | deny | `"deny"` |
| R-5 | No `agent_type` (main session) | absent | ask | `"ask"` |
| R-6 | Explicit `tech-lead` in payload | `"tech-lead"` | ask (validator rejects; falls to unknown branch) | `"ask"` |
| R-7 | Unrecognized string | `"bogus-agent"` | ask | `"ask"` |
| R-8 | Librarian — Read target | `"librarian"`, `file_path` = path not CUSTOMER_NOTES | proceed | (empty stdout) |
| R-9 | Librarian — bash grep CUSTOMER_NOTES | `"librarian"`, Bash `grep CUSTOMER_NOTES.md` | proceed | (empty stdout; read-not-write) |

Assertion logic for new `allow` / `deny` cases mirrors the existing `fire`
(ask) pattern: check `stdout` for the `permissionDecision` value string.

### D.2 Regression against existing read-vs-write suite

All existing cases in `tests/hooks/test-customer-notes-guard.sh` (read-only
commands: `grep`, `cat`, `sed -n`, heredoc reads, `python3 -c open(path).read()`,
etc.) must pass unchanged. The new role-resolution path is only reached
*after* `_tool_input_touches_customer_notes()` returns True AND
`_is_write_to_customer_notes()` is satisfied — the same gate as before.
None of the existing test payloads include `agent_type`, so they fall into
the `role is None → ask` branch and produce the same output as today.

Run the full existing suite as part of the v1.5.3 pre-commit gate with no
expected failures.

### D.3 Unit tests for shared `role_resolver.py`

Create `tests/hooks/test-role-resolver.py` (pytest):

```python
from scripts.hooks.lib.role_resolver import resolve_subagent_role

def test_agent_type_librarian():
    role, src = resolve_subagent_role({"agent_type": "librarian", "tool_input": {}})
    assert role == "librarian"
    assert src == "agent_type"

def test_agent_type_tech_lead_rejected():
    role, _ = resolve_subagent_role({"agent_type": "tech-lead", "tool_input": {}})
    assert role is None  # validator rejects self-push

def test_subagent_role_alias():
    role, src = resolve_subagent_role({"subagent_role": "architect", "tool_input": {}})
    assert role == "architect"
    assert src == "subagent_role"

def test_no_identity():
    role, src = resolve_subagent_role({"tool_input": {}})
    assert role is None and src is None

def test_sme_slug():
    role, src = resolve_subagent_role({"agent_type": "sme-billing", "tool_input": {}})
    assert role == "sme-billing"

def test_bogus_role():
    role, _ = resolve_subagent_role({"agent_type": "bogus-agent", "tool_input": {}})
    assert role is None
```

### D.4 Integration test — verify `agent_type` presence on spawned subagent Write events

This is the load-bearing assumption: that when a Claude Code subagent
issues a Write/Edit call, the `PreToolUse` hook event payload contains
`agent_type` equal to the role name used at spawn time.

**Smallest spawn-based test design:**

1. Create a throwaway hook script `tests/hooks/integration/capture-hook-event.py`
   that reads the hook event from stdin, writes it as JSON to a temp file
   (path in env var `CAPTURE_OUT`), and returns `permissionDecision: "allow"`.

2. Create a minimal subagent agent file `tests/hooks/integration/test-writer.md`
   (role: `test-writer`, tools: Write) with a single task: write one line to
   a temp file.

3. Create a shell driver `tests/hooks/integration/test-agent-type-presence.sh`:
   - Overrides the `Write` hook to `capture-hook-event.py` for the test run.
   - Spawns the `test-writer` subagent via the Claude Code harness (or a
     minimal harness stub that calls the hook on a synthetic Write event with
     the subagent's role in `agent_type`).
   - After the subagent completes, reads the captured event and asserts
     `agent_type == "test-writer"`.

**Note on hermetic vs live testing:** A true end-to-end spawn test requires
the Claude Code harness to be running, which is not available in CI. The
integration test must therefore be classified as a **manual / environment-
gated** check, run once per new harness version to re-validate the
assumption. Its purpose is to fail loudly if a harness update removes
`agent_type` from hook payloads. The CI unit-test suite (D.1–D.3) remains
hermetic and is the blocking gate.

Document the assumption explicitly in `scripts/hooks/lib/role_resolver.py`
docstring:

> **Load-bearing assumption (validated 2026-06-16 on Claude Code harness):**
> When a specialist subagent is spawned by name, the `PreToolUse` hook
> event payload contains `agent_type` equal to the spawned role name. If the
> harness removes this field, `resolve_subagent_role()` returns `(None, None)`
> for all spawned agents, which causes `customer-notes-guard.py` to fall back
> to `"ask"` for all writes — a regression to the pre-v1.5.3 behavior but
> not an expansion of the attack surface. The Antigravity fallback
> (`ANTIGRAVITY_CONVERSATION_ID` + `docs/pm/active-dispatches.json`)
> provides partial mitigation for Antigravity harnesses. Re-run integration
> test D.4 after any harness version upgrade.

v1.5.1 / v1.5.2 should treat upgrade to v1.5.3 as a security-critical
patch (Hard Rule #1 class violation).

### E.3 Downstream migration notes

Projects upgrading from v1.5.0–v1.5.2 to v1.5.3 receive:

1. **`settings.json` permissions block (automatic via upgrade script).**
   The `"permissions": {"defaultMode": "acceptEdits"}` addition is a
   net-new key and does not conflict with any existing project customization
   of settings.json. The upgrade script should merge it with a `jq`-style
   deep-merge rather than overwriting the whole file, preserving any
   project-local hook additions.

2. **New `scripts/hooks/lib/role_resolver.py` (automatic — new file).**
   No migration action required. Projects that import from hook lib modules
   directly (unusual) should verify no naming collision.

3. **`customer-notes-guard.py` behavior change (explicit migration note).**
   After upgrade, librarian-dispatched `CUSTOMER_NOTES.md` writes proceed
   silently (no human prompt). Any downstream project that relied on the
   `"ask"` prompt as a workflow step (e.g., using it as an informal review
   trigger) must establish an alternative review mechanism. The recommended
   alternative is tech-lead review of the librarian dispatch brief before
   sending it, per the template contract.

4. **No schema or data migration.** `CUSTOMER_NOTES.md` format is
   unchanged. The `docs/pm/active-dispatches.json` Antigravity fallback is
   unchanged.

## F. Security Analysis Cross-Reference and Customer-Facing Open Question

### F.1 Cross-reference

The security assurance analysis for this change is being authored in
parallel at:

```
docs/security/sa-v1.5.3-hr1-permission-leak.md
```

That document covers: threat model for the `agent_type` role gate (spoofing
risk, payload integrity, trust boundary), ISO/IEC 15026-2:2022 residual-risk
framing, and the sign-off required by Hard Rule #7. This design document
does not duplicate that analysis. Any design decision in §§ B–C above that
implicates the trust boundary (particularly: what happens if `agent_type`
can be forged in a hook event) must be reconciled against the security
analysis before implementation starts.

### F.2 Customer-facing open question

**Converting the `customer-notes-guard.py` gate from `"ask"` to
role-deterministic removes the only human checkpoint on `CUSTOMER_NOTES.md`
write integrity.**

Under the pre-v1.5.3 model, every write to customer truth — authorized or
not — surfaces a human confirmation. Under the v1.5.3 model, librarian
writes proceed without a prompt. The assurance that the written content is
correct shifts entirely onto:

- the `agent_type` role gate (hook-enforced: only `librarian` gets `allow`),
- tech-lead review of the dispatch brief that instructs librarian (Hard
  Rule #8: tech-lead orchestrates what librarian records), and
- the advisory content-findings check (still runs for librarian writes,
  emitted to stderr/audit log).

This is a trade-off between **workflow friction** (the pre-v1.5.3 "ask"
that fires on every librarian dispatch) and **trust-model clarity** (the
v1.5.3 role gate that eliminates false-positive customer prompts but relies
on the hook's `agent_type` assumption).

*This is the single open question that requires customer decision before
implementation commits.*
