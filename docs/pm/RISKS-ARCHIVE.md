## Response plan details (for material risks)

### R-M0-1 Context bloat persists

- Detailed response plan: keep G0 and G1 as blocking gates; reject new default-live surfaces without measurement fields or context-cost justification.
- Contingency: split oversized live artifacts into live/evidence/archive surfaces before accepting later milestone work.
- Secondary risks introduced by the response: more support artifacts can create confusion unless authority classes stay current.
- Cost of the response: PM review time and specialist review gates; no new runtime dependency.

### R-M0-2 Authority drift

- Detailed response plan: maintain artifact authority class records and require generated/manual artifacts to cite canonical inputs.
- Contingency: quarantine any generated or manual artifact that introduces unapproved binding rules.
- Secondary risks introduced by the response: review overhead for small docs changes.
- Cost of the response: architect/code-reviewer review time.

### R-M0-3 Prompt-compiler drift

- Detailed response plan: keep compiler implementation out of M0/M1 and review runtime candidates as generated, subordinate artifacts.
- Contingency: block G1 until drift is corrected or an explicit exception is recorded.
- Secondary risks introduced by the response: slower path to automated generation.
- Cost of the response: prompt-regression and code-review effort.

### R-M0-4 Model-routing volatility

- Detailed response plan: defer routing implementation and avoid binding future exact model IDs in M0/M1 acceptance artifacts.
- Contingency: re-plan routing assumptions at the later routing gate.
- Secondary risks introduced by the response: future routing work may need fresh research.
- Cost of the response: release-engineer/researcher verification later.

### R-M0-5 Archive traceability loss

- Detailed response plan: require append-only archives, live tombstones, source-file headers, and evidence links before any live rows are removed.
- Contingency: restore missing evidence from git history and block G1 until traceability is repaired.
- Secondary risks introduced by the response: archive files can grow, but they are not default live reads.
- Cost of the response: archival script and review effort.

### R-M0-6 Runtime-compaction rule loss

- Detailed response plan: require prompt-regression coverage and preservation review for hard rules, local supplement checks, escalation behavior, role authority, and customer-interface ownership.
- Contingency: reject or regenerate affected runtime candidates and record exceptions only when safe.
- Secondary risks introduced by the response: some reduction targets may need exceptions.
- Cost of the response: qa/code-reviewer review time.

## Review cadence

`project-manager` reviews the register at least every milestone. High-
score open risks reviewed in the **first session of each calendar
week**. Every review bumps `Last reviewed`.

Cadences in this framework are **session-anchored, run-once** — the
agent only executes when the customer opens a session, so a cadence
expressed as "weekly" means "in the first session on or after the
week boundary." Missed weeks do not accumulate; the next session runs
the review once, not twice. See `CLAUDE.md` § "Time-based cadences."
