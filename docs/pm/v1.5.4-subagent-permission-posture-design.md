## 2. Design Options for `permissions.allow`

Claude Code `permissions.allow` entries take the form
`"Bash(<prefix>:*)"` where the prefix is matched against the beginning of
the command string. They remove the interactive human prompt for matching
commands; they do NOT suppress deny-hooks.

The three deny-hooks relevant here are:

- `customer-notes-guard.py` — gates CUSTOMER_NOTES.md Bash writes
- `tech-lead-authoring-guard.py` — gates off-allow-list path writes by tech-lead
- `handoff-pre-tool-gate.py` — gates path-scope violations on active handoffs

All three still fire and can return `permissionDecision: "deny"` regardless
of any `permissions.allow` entry. A `permissions.allow` only removes the
human checkpoint; it does not turn a hook deny into a pass.

### Option A — Targeted allow-list

Allow only the specific command prefixes that routinely fire and are judged
safe. Everything else still prompts (and wedges on remote control).

Concrete JSON:

```json
"permissions": {
  "defaultMode": "acceptEdits",
  "allow": [
    "Bash(git diff*)",
    "Bash(git log*)",
    "Bash(git status*)",
    "Bash(git show*)",
    "Bash(git add*)",
    "Bash(git commit*)",
    "Bash(git push*)",
    "Bash(git tag*)",
    "Bash(git fetch*)",
    "Bash(git checkout*)",
    "Bash(git switch*)",
    "Bash(git stash*)",
    "Bash(git merge*)",
    "Bash(git rebase*)",
    "Bash(git reset*)",
    "Bash(git worktree*)",
    "Bash(grep *)",
    "Bash(rg *)",
    "Bash(cat *)",
    "Bash(ls *)",
    "Bash(find *)",
    "Bash(head *)",
    "Bash(tail *)",
    "Bash(wc *)",
    "Bash(diff *)",
    "Bash(python3 *)",
    "Bash(pytest*)",
    "Bash(bash scripts/*)",
    "Bash(gh issue*)",
    "Bash(gh pr*)",
    "Bash(gh release*)"
  ]
}
```

**Weakness of Option A (critical):** See §4. This list will not cover
every command a specialist runs. Any uncovered command still prompts and
wedges on remote control. Specialists also compose commands with pipes,
`&&`, variable interpolation, and shell built-ins (e.g., `cd dir && pytest`,
`[ -f foo ] && bash scripts/...`). Prefix matching does not handle these
reliably; `Bash([ -f*)` or `Bash(cd*)` would need to be added and would
be effectively unlimited in scope. This approach is a partial mitigation
that reduces wedge frequency but does not eliminate it.

### Option B — Broad `Bash(*)`

Allow all Bash commands unconditionally. Hook-based deny decisions (the
three deny-hooks enumerated above) remain the sole enforcement layer.

Concrete JSON:

```json
"permissions": {
  "defaultMode": "acceptEdits",
  "allow": [
    "Bash(*)"
  ]
}
```

This is the simplest change that fully eliminates the remote-control wedge
for Bash. No specialist Bash call ever prompts. The only safety net is
the hook layer.

## 3. Security Tradeoff Analysis

### 3.1 Threat model baseline

Per the customer's authoritative context and the framework's standing threat
model: specialists are the team's own trusted agents. The concern is
accidental rule violations (a specialist writing to the wrong path, a
misrouted customer-notes write), not an adversarial subagent. This is the
same trust model under which `SWDT_AGENT_PUSH` and `agent_type` residuals
were accepted in v1.5.2.

This threat model matters for the analysis below. If the threat model were
adversarial subagents, both options A and B would require much tighter
scrutiny. Under the trusted-specialist model the residual risk from either
option is primarily misconfiguration/accident rather than attack.

### 3.2 What Option A auto-allows without human review

All commands matching the listed prefixes execute without a prompt. The
allow entries are coarse:

- `Bash(bash scripts/*)` covers every script in `scripts/`, including
  `scripts/setup-github-labels.sh` (remote-mutating) and `upgrade.sh`
  (destructive if misused). A specialist that mistakenly invokes one of these
  would not be stopped by a permission prompt.
- `Bash(python3 *)` covers any Python invocation, including one that opens
  a file for write. The hook layer (tech-lead-authoring-guard) is the guard;
  if the target path is off the allow-list and no escape hatch is active, the
  hook denies. But a Python invocation that targets an allowed path or
  `/tmp/` gets through.
- `Bash(gh release*)` auto-allows release creation and upload without a
  human checkpoint. Release-engineer is the expected caller; an accidental
  dispatch from another agent would fire the hook deny only if a hook
  specifically gates it (none currently does for `gh` calls).
- `Bash(git push*)` auto-allows force-push if the agent appends
  `--force`. No current hook checks for `--force` in git commands.
- The targeted list provides **false assurance**: it looks more controlled
  than Option B but leaves the same meaningful gaps. A `git push --force`,
  `rm -rf`, or `curl` that slips through a prefix pattern in a composed
  command is not blocked.

### 3.3 What Option B auto-allows without human review

Everything. `rm -rf`, `curl <url>`, `git push --force`, `gh release create`
for a wrong tag, arbitrary shell scripts, network exfiltration if an agent
were compromised. The only enforcement is:

- Hook layer (three deny-hooks): these cover path-scoped writes and
  CUSTOMER_NOTES.md writes. They do NOT cover `rm`, `git push --force`,
  arbitrary `curl`, or GitHub CLI calls.
- Agent contract constraints (role prose): agents are instructed what not
  to do, but instruction is not enforcement.

Under the trusted-specialist threat model the practical risk is accidental
misuse — a `rm` or `git push --force` emitted by a confused agent, not by a
malicious one. The consequence is reversible in most cases (git history
persists; GitHub has a recovery path for deleted releases) but can be
disruptive.

### 3.4 Command families: genuinely safe vs should-still-prompt

| Family | Option A targeted | Option B broad | Genuine risk | Assessment |
|---|---|---|---|---|
| `git diff/log/status/show` | Allowed | Allowed | Negligible (read-only) | Safe to allow |
| `git add/commit` | Allowed | Allowed | Low (reversible; hook-enforced path scope) | Safe under trusted-specialist model |
| `git push` (non-force) | Allowed | Allowed | Medium (remote state mutation, not easily undone for tags) | Acceptable under trusted-specialist model; force-push is the dangerous sub-case |
| `git push --force` / `--delete` | Allowed via `git push*` | Allowed | High (irreversible remote mutation) | Neither option protects against this |
| `grep/rg/cat/ls/find` | Allowed | Allowed | Negligible | Safe |
| `pytest` / `python3` (read-only) | Allowed | Allowed | Negligible | Safe |
| `python3` (write to allowed path) | Allowed | Allowed | Low; hook guards apply | Acceptable |
| `bash scripts/lint-*.sh` / check / audit | Allowed via `bash scripts/*` | Allowed | Negligible | Safe |
| `bash scripts/pre-release-gate.sh` | Allowed via `bash scripts/*` | Allowed | Low | Acceptable |
| `bash scripts/upgrade.sh` | Allowed via `bash scripts/*` | Allowed | Medium (scaffold mutation) | Neither option protects |
| `gh issue/pr view/list` | Allowed | Allowed | Negligible | Safe |
| `gh pr create/merge` | Allowed | Allowed | Medium (remote PR state) | Acceptable under trusted model |
| `gh release create` | Allowed | Allowed | High if wrong tag | Neither option protects |
| `rm` / `rm -rf` | NOT in targeted list | Allowed | High (destructive) | Option A provides a marginal barrier here |
| `curl` / `wget` | NOT in targeted list | Allowed | High (network exfil if model compromised) | Option A provides a marginal barrier |
| `git push --force` | Allowed via `git push*` | Allowed | High | Neither option protects |
| `bash scripts/setup-github-labels.sh` | Allowed via `bash scripts/*` | Allowed | Medium | Neither option protects once `bash scripts/*` is in the list |

**Key finding:** Once `Bash(bash scripts/*)`, `Bash(git push*)`, and
`Bash(gh release*)` are in the targeted list — and they must be, because
specialists routinely need them — the practical difference between Option A
and Option B collapses for the highest-risk commands. The commands that are
genuinely dangerous (`rm -rf`, arbitrary `curl`) are the primary thing
Option A adds marginal protection against.

Under the trusted-specialist threat model, `rm` and `curl` are not attack
vectors; they are commands an agent might emit accidentally. The question
is whether removing the human checkpoint on those specific commands is
acceptable. The framework's prior decisions (accepting `SWDT_AGENT_PUSH`
residuals, accepting `agent_type` trust) suggest the answer is yes.

### 3.5 Where to draw the line

Given the trusted-specialist threat model, the line should be drawn at:

- **No human checkpoint needed:** all read-only commands; git read ops;
  `pytest` / `python3`; lint/check scripts; `gh` read ops; `git add/commit`;
  non-force `git push`; `bash scripts/` scripts not known to be destructive.
- **Human checkpoint arguable:** `rm`, `curl`/network commands, `git push --force`,
  `gh release create` against wrong tags. None of these are currently protected
  by hooks, and Option A's prefix matching does not reliably exclude them
  from composed commands anyway.

The honest assessment is that neither option provides meaningful protection
against the highest-risk sub-cases once `bash scripts/*` and `git push*`
are in the list. The practical security delta between A and B is narrow
under this threat model.

## 4. Completeness Honesty: The Targeted List Will Not Cover Everything

This must be stated plainly.

A targeted `permissions.allow` list (Option A) addresses the most common
command patterns. It does NOT cover:

- Shell built-ins and compound forms: `cd /path && pytest`, `[[ -f x ]] && bash ...`,
  `for f in ...; do cat $f; done`, `export VAR=x && python3 ...`
- Commands a specialist invokes that were not anticipated when the list was
  written. New scripts added to `scripts/` automatically become uncovered.
- Commands with unusual argument ordering where the prefix match fails.
- Subshell invocations: `$(bash scripts/foo.sh)`, `bash -c "..."`.
- Edge cases in agent behavior: an agent that issues `echo hello | bash` or
  `env -i bash scripts/foo.sh` will not match the `Bash(bash scripts/*)` prefix.

Every uncovered Bash command still prompts, and on remote control that
prompt does not surface — the subagent hangs indefinitely until the session
times out or the operator restarts it. The wedge problem is reduced but
not eliminated.

**Only two approaches fully eliminate the wedge:**

1. `Bash(*)` in `permissions.allow` (Option B), which pre-authorizes
   everything.
2. The upstream harness fix (#356): subagents inherit the main-session
   bypass mode. This is not in the scaffold's control.

Any targeted list is a partial mitigation that reduces wedge frequency
in proportion to how well it covers observed command patterns. For a
framework used across diverse downstream projects with project-specific
scripts, the coverage will be imperfect from day one and will degrade
as projects add scripts.

## 5. Verdict and Recommended `permissions.allow`

### 5.1 Recommendation: Option B — `Bash(*)`

**Recommended configuration:**

```json
"permissions": {
  "defaultMode": "acceptEdits",
  "allow": [
    "Bash(*)"
  ]
}
```

Rationale:

1. **The targeted list (Option A) is false precision.** Once `bash scripts/*`
   and `git push*` are included — which they must be for routine operations —
   the meaningful protection Option A adds over Option B is limited to `rm`
   and bare `curl`. Under the trusted-specialist threat model, accidental
   `rm` is an accepted residual risk comparable to the already-accepted
   `SWDT_AGENT_PUSH` and `agent_type` trust residuals.

2. **Option A does not solve the wedge problem; it only reduces it.**
   Compound commands, shell built-ins, and unanticipated scripts remain
   uncovered. Remote-control deployments will still wedge on those cases.
   If the customer is operating under remote control, a partial fix that
   still wedges on a compound `git` call is not meaningfully better than
   the current state.

3. **The hook layer is the real guard.** The three deny-hooks enforce
   path scope, authoring rules, and CUSTOMER_NOTES.md integrity. They run
   regardless of `permissions.allow`. A `permissions.allow: Bash(*)` removes
   the human checkpoint but does not disable hook-based denies. The
   framework's defense-in-depth is hook-first, not prompt-first.

4. **Precedent: `defaultMode: acceptEdits`.** v1.5.3 already accepted that
   all Edit/Write/MultiEdit tool calls by subagents proceed without a human
   prompt. `Bash(*)` extends the same principle to Bash. The threat model
   did not change; only the tool family did.

5. **Drift risk of maintaining a targeted list.** The targeted list must be
   updated every time a new script is added to `scripts/`, every time an
   agent gains a new Bash pattern, and for every downstream project's
   custom scripts. The maintenance burden is ongoing, and a missed entry
   wedges production use. `Bash(*)` has no drift risk.

### 5.2 Residual risks under Option B (accepted under trusted-specialist model)

- `rm` / `rm -rf` by an agent that misidentifies a target: reversible in
  most cases via git history.
- `git push --force` by a confused agent: recoverable on GitHub for most
  branch types; protected-branch rules on the remote add a second layer.
- `curl` / network calls: no hook currently detects these. The trusted-
  specialist model accepts this given that agents have no motivation to
  exfiltrate and no external commands instruct them to do so.
- `gh release create` against a wrong tag: recoverable; release-engineer
  contract requires `pre-release-gate.sh` pass before tagging.

None of these risks are new to v1.5.4; they exist today and would fire a
human prompt that the operator cannot see on remote control, which is
arguably worse than allowing the command (a hidden prompt wedges the session
entirely, losing all subsequent work).

### 5.3 Does this warrant `security-engineer` sign-off as a v1.5.4 release?

Hard Rule #7 requires `security-engineer` sign-off for releases touching
authentication, authorization, secrets, PII, or network-exposed endpoints.
This change modifies the **permission posture** of the scaffold but does not
touch auth-code, secrets handling, PII paths, or network-exposed surface
area in the sense of the rule. The deny-hooks (which enforce auth/authz
rules at the tool level) are unchanged.

Assessment: this change sits at the boundary. It is a security-relevant
configuration change (it removes human checkpoints from a broad tool class)
but it is not a change to auth or authz logic. The deny-hooks that enforce
the authoring rules, customer-notes gate, and handoff path scope continue to
operate unchanged. The permission posture change is therefore analogous to
the v1.5.3 `defaultMode: acceptEdits` change, which shipped without a
separate Hard Rule #7 flag.

**Recommendation:** route the `Bash(*)` decision through `tech-lead` for a
customer sign-off on the threat-model acceptance (specifically: is the
customer comfortable that accidental `rm` / `git push --force` from a
confused agent is an accepted residual risk under the trusted-specialist
model?). Record the customer ruling in `CUSTOMER_NOTES.md`. This is
customer-preference territory, not a Hard Rule #7 security sign-off
requirement. However, if the customer flags the change as security-sensitive
for their deployment context, apply Hard Rule #7 in full.

Projects upgrading from v1.5.3 to v1.5.4 receive `allow: ["Bash(*)"]` in

own `permissions.allow` entries should verify no semantic conflict. Projects
that rely on interactive Bash prompts as a workflow step (uncommon; the
`defaultMode: acceptEdits` floor already suppresses Edit/Write prompts)
must establish an alternative control. No data migration required.

### 8.1 Exact `permissions` block (copy-ready)

```json
"permissions": {
  "defaultMode": "acceptEdits",
  "allow": [
    "Bash(*)"
  ],
  "deny": [
    "Bash(git push --force*)",
    "Bash(git push -f *)",
    "Bash(git push -f)",
    "Bash(git push --delete*)"
  ]
}
```

No `ask` entries. Rationale for the empty `ask` list is in §8.4.

---

### 8.2 Deny-list entries: rationale and legitimate-work impact

**Entry: `Bash(git push --force*)`, `Bash(git push -f *)`, `Bash(git push -f)`**

What it blocks: force-push to any remote ref via the long flag (`--force`) or short flag (`-f`). A force-push rewrites remote branch history non-reversibly (protected branches on GitHub add a second layer, but unprotected branches and tags do not benefit from that).

Why it is never legitimate: no agent contract in `.claude/agents/` includes force-push as an authorized operation. `release-engineer` uses `git push` (non-force) and `git tag` followed by `git push --tags` (also non-force). `software-engineer` pushes feature branches non-force. Force-push is an emergency human operator action, not a specialist Bash action. There is no script in `scripts/` that invokes `git push --force` or `git push -f`.

Legitimate-work impact: none found. A full scan of `.claude/agents/*.md` and `scripts/**/*.sh` found no force-push invocation. Blocking this is safe.

**Entry: `Bash(git push --delete*)`**

What it blocks: deletion of a remote ref (branch or tag) via `git push --delete <remote> <ref>`. Tag deletion on the remote is irreversible in the same way as force-push.

Why it is never legitimate: no agent contract or script invokes `git push --delete`. Release-engineer creates tags; it does not delete them. Branch deletion goes through `gh pr merge` (which GitHub handles) or is a human operator action. There is no scenario in the framework's workflow where a specialist legitimately needs to delete a remote ref directly.

Legitimate-work impact: none found. Safe to deny.

**Why `rm` and `curl` are NOT in the deny list**

`rm` (including `rm -rf`) appears extensively throughout `scripts/*.sh` as legitimate temp-dir cleanup in `trap ... EXIT` handlers. Concrete examples:

- `pre-release-gate.sh`: `trap 'rm -rf "$GATE_TEMP_ROOT"' EXIT` — cleans its own temp working directory.
- `smoke-test.sh`, `stepwise-smoke.sh`: `trap 'rm -rf "$tmp"' EXIT` — cleans mktemp-created directories.
- `worktree-teardown.sh`: `rm -rf "$WORKTREE_PATH"` — scoped by a safety guard to `/tmp/agent-*` paths only; the script refuses any path not matching that pattern.
- `generate-fixture-snapshots.sh`, `lib/gate-migrations.sh`, `lib/gate-tags.sh`: `rm -rf` on fixture and worktree temp directories inside `$workdir` and `$tmpdir` variables.
- `check-spdx.sh`, `lint-questions.sh`, `lint-agent-model-routing.sh`, `lint-routing.sh`: `rm -f` on mktemp temp files.
- `upgrade.sh`: `rm -f` on conflict files, block artefacts, and merge logs; `rm -rf` on worktree clones inside `$workdir`.

A `deny` on `Bash(rm*)` or `Bash(rm -rf*)` would break every one of these scripts. Because the pattern matcher operates on the command string as presented to Bash, `rm -rf "$GATE_TEMP_ROOT"` embedded in a trap string would actually be invoked as a separate shell command and would match a leading-`rm` deny — blocking `pre-release-gate.sh`, `smoke-test.sh`, and other release-critical scripts. This is not an acceptable breakage.

`curl` does not appear in any script in `scripts/**/*.sh` as an actual invocation (one comment in `stepwise-smoke.sh` references a historical manual `curl` recovery step but does not invoke it). No agent contract mentions `curl` as an authorized Bash operation. However, blocking `Bash(curl*)` is not included in the deny list because: (a) there is no current usage to protect against, making the entry provide only marginal theoretical value; (b) downstream projects may legitimately add `curl`-based health checks, download helpers, or webhook-trigger scripts, and a framework-level deny would silently break those without any per-project override path.

The honest position: for `rm` and `curl`, the trusted-specialist threat model (established in §3.1) is the appropriate control — not a coarse prefix deny that either breaks legitimate work (`rm`) or punishes downstream extensibility (`curl`).

### 8.3 Honest limitations of this deny list

**Coarse prefix matching only.** Claude Code `permissions.deny` patterns are matched against the leading portion of the command string as a glob. The matching engine does not parse shell syntax; it does not see inside `&&`-chained commands, pipes, or subshells.

Concrete evasion scenarios:

- `cd /tmp && git push --force origin main` — the leading token is `cd`, not `git push --force`. The deny pattern does not match. The force-push executes.
- `bash -c "git push --force origin main"` — the leading token is `bash`, not `git push`. The deny does not trigger.
- `git push origin main --force` — argument order matters: `Bash(git push --force*)` matches `git push --force...` but not `git push origin main --force` where `--force` appears after the refspec. **This is a real gap in the deny list as written.**

The third case is significant: `git push origin <refspec> --force` is a natural form an agent might emit. To close it partially, the deny list would need `Bash(git push*--force*)` — but this is not standard glob syntax in Claude Code's permission system; only prefix globs are supported. This gap cannot be fully closed with the available pattern language.

**The deny list is a thin backstop, not a containment boundary.** It catches the most obvious textbook form of force-push (`git push --force <remote>`) and tag/branch deletion. It does not catch compound invocations, reversed argument order, or bash-wrapped commands. Under the trusted-specialist threat model this is acceptable because: (1) specialists do not intentionally force-push; the deny is protection against confused/misdirected agent behavior, not adversarial bypasses; (2) GitHub protected-branch rules on the remote provide a second layer for main/protected branches; (3) git reflog and GitHub's ref deletion recovery provide third-layer recoverability.

**`allow: ["Bash(*)"]` takes precedence over `deny` only where there is no deny match.** The semantics of Claude Code's `permissions` system are: `deny` entries are evaluated before `allow`; a matching `deny` blocks regardless of `allow`. The `Bash(*)` allow does not override a `deny` match. This is the intended hybrid behavior.

---

### 8.4 Why `ask` is empty

The customer constraint is explicit: on remote-control interfaces, an `ask` decision does not surface — it wedges the operator. The entire motivation for this change is eliminating wedges. Adding `ask` entries would recreate the wedge for those patterns. Since the deny list covers only patterns that are never legitimate (force-push, ref deletion), there is no pattern for which "prompt a human" is the right response on remote control: either it is a legitimate operation (allow it) or it is never legitimate (deny it outright).

If a future pattern arises where human review is genuinely required and remote-control wedging is acceptable, that pattern belongs in neither this `permissions` block nor in an automated flow — it belongs in an explicit operator runbook step outside the agent loop.

---

### 8.5 Residual-risk statement (for fw-adr-NNNN)

> The hybrid permission posture (`allow: ["Bash(*)"]` with a small `deny` list covering force-push and remote-ref deletion) eliminates the remote-control Bash wedge while blocking the two git operations that are both catastrophic and never legitimately emitted by a framework specialist. The deny list is a thin backstop constrained by Claude Code's coarse prefix-glob matcher: compound commands (`cd /path && git push --force ...`), bash-wrapped invocations (`bash -c "git push --force ..."`), and reverse-argument forms (`git push origin main --force`) evade the deny patterns. The guard's value is therefore limited to catching the most naive/accidental form of these operations by a confused agent, not to containing deliberate misuse. Under the trusted-specialist threat model established in v1.5.2 and reiterated in §3.1 of this document — where specialists are the team's own agents with no adversarial motivation — this thin backstop is judged sufficient. Legitimate `rm` usage throughout the release scripts precludes a `deny` on `rm`; that residual risk is accepted as equivalent to the `SWDT_AGENT_PUSH` and `agent_type` trust residuals previously accepted. The primary enforcement layer remains the hook-based deny system (customer-notes-guard, tech-lead-authoring-guard, handoff-pre-tool-gate), which operates independently of `permissions` settings and cannot be bypassed by any `allow` entry.

## 9. Final Union Deny List and "Destructive Ops = Tech-Lead Duty" Design

**Status:** Customer-ratified. Customer directive (2026-06-23): "all destructive work should run through the tech lead as a declared duty of the tech lead."
**Author:** security-engineer

---

### 9.1 Final union `deny` array (copy-ready for both scaffold and meta `settings.json`)

The union starts from the meta's existing deny list and the scaffold's v1.5.4 deny list, deduplicates, and adds a small set of entries that satisfy all three admission criteria: (a) genuinely destructive with no reversible path, (b) expressible as a leading-prefix glob in Claude Code's permission system, and (c) not present in any agent contract or script as a legitimate subagent invocation.

```json
"deny": [
  "Bash(sudo *)",
  "Bash(dd *)",
  "Bash(mkfs *)",
  "Bash(shred *)",
  "Bash(truncate *)",
  "Bash(rm *)",
  "Bash(chmod 777 *)",
  "Bash(git reset --hard*)",
  "Bash(git clean -f*)",
  "Bash(git checkout -- *)",
  "Bash(git restore --*)",
  "Bash(git branch -D *)",
  "Bash(git push --force*)",
  "Bash(git push --force-with-lease*)",
  "Bash(git push -f *)",
  "Bash(git push -f)",
  "Bash(git push --delete*)",
  "Bash(git push --mirror*)"
]
```

**Ordering rationale:** privilege escalation first; disk/storage destruction second; destructive local git ops third; destructive remote git ops fourth. Within each group, entries are ordered from most dangerous to most specific variant.

---

### 9.2 Entry-by-entry rationale for additions beyond the starting union

**`Bash(shred *)`**

What it blocks: secure file overwrite-and-deletion. More destructive than `rm` — intentionally defeats recovery tools. No agent contract or script in `scripts/**/*.sh` invokes `shred`. No framework workflow requires it. The meta deny already blocks `Bash(rm *)` for the meta project; `shred` is the more dangerous sibling and belongs in both lists.

Legitimate-work impact: none found. Safe to add.

**`Bash(truncate *)`**

What it blocks: in-place file zeroing without removal. A truncated file loses its content but retains its inode; git will not recover the content unless it was staged. No script in `scripts/**/*.sh` calls `truncate`. Agents have no authorized use for it. The destructive effect (silent data loss on a tracked file) is non-obvious and not caught by `rm`-scoped recovery advice.

Legitimate-work impact: none found. Safe to add.

**`Bash(git push --force-with-lease*)`**

What it blocks: force-push variant that checks the remote ref before overwriting. It is less dangerous than `--force` (a concurrent push will cause it to fail safely) but it still rewrites remote history non-reversibly once it executes. No agent contract or script invokes force-with-lease; the framework's push operations are all non-force. The customer's principle is "all destructive ops via tech-lead" — any remote history rewrite, regardless of safety variant, qualifies as destructive. Adding this closes the most obvious gap left by denying `--force` alone.

Legitimate-work impact: none found. Safe to add.

**`Bash(git push --mirror*)`**

What it blocks: a mirror push replaces the entire remote — all branches, tags, and refs — with the local state. It deletes remote refs that do not exist locally. This is the most destructive single git push variant available and is categorically an operator action, never a specialist action. No agent contract or script invokes it.

Legitimate-work impact: none found. Safe to add.

**`Bash(rm *)` — retained in the union for both contexts**

In the scaffold context, §8.2 established that `rm` cannot be denied because `bash scripts/*` is in the scaffold's `allow: ["Bash(*)"]` and the release scripts use `rm -rf` in trap handlers. That analysis remains correct for the scaffold. However, the meta project does NOT have `Bash(*)` in its allow; it has a narrow targeted allow list that does not include `bash scripts/*`. Meta subagents therefore have no legitimate path to `rm` via an allowed script, and the meta's existing `Bash(rm *)` deny is correct and should be retained.

The union deny list above applies to both files. For the scaffold, the `Bash(rm *)` entry will be evaluated against the `allow: ["Bash(*)"]` entry. Per Claude Code's documented semantics, `deny` takes precedence over `allow` for matching commands. This means `rm` will be denied in the scaffold as well as the meta. This is a change from the §8.2 conclusion and is a direct consequence of the customer's "all destructive ops via tech-lead" principle. The effect on the scaffold's release scripts is: any script that calls `rm` from a trap handler will be blocked if invoked as a Bash command directly by a subagent. The mitigations are: (1) tech-lead runs these scripts from the main session (which is bypassPermissions and not subject to deny), or (2) the scripts are refactored to avoid `rm` in subagent-callable contexts. This is a known trade-off the customer's principle accepts.

### 9.3 Candidates considered and rejected

**`Bash(curl *)` and `Bash(wget *)`**

Rejected for the same reason as in §8.2: no current script uses them, and downstream projects may legitimately add curl-based health checks, webhook triggers, or asset download steps. A framework-level deny would silently break those without an override path. The trusted-specialist threat model (§3.1) applies here; the customer's principle targets destructive file/git operations, not network calls.

**`Bash(kill *)` and `Bash(pkill *)`**

Process termination is disruptive but not destructive in the data-loss sense. Agents have no authorized use for these, but they are also unlikely accidental emissions and are recoverable (the process can be restarted). Marginal value; rejected as noise.

**`Bash(git tag -d *)` and `Bash(git push --tags)`**

Tag deletion (`git tag -d`) is local and recoverable (the tag can be recreated from the SHA). `git push --tags` is a normal release operation used by `release-engineer`. Neither qualifies under criterion (a) — genuinely destructive with no reversible path — or criterion (c) — not in legitimate subagent use. Rejected.

### 9.4 Duty statement for scaffold `CLAUDE.md` — placement and text

**Placement recommendation: Hard rules, as a new rule after existing Rule #12 (parallel working-tree isolation).**

Rationale for Hard rules over routing/duties: the routing section describes how work is dispatched between agents. This constraint is about what agents at the execution layer are prohibited from doing regardless of dispatch context. Hard Rule #8 establishes that tech-lead does not author production artifacts directly; this rule is the parallel constraint at the Bash execution layer — subagents do not execute destructive shell commands directly. These are structural constraints on agent capability, not routing preferences. Both belong in Hard rules, alongside the other categorical prohibitions.

**Proposed Hard Rule text (for scaffold `CLAUDE.md`):**

> **Destructive Bash operations are a tech-lead duty.** The `deny` list in `.claude/settings.json` blocks the most obvious destructive Bash commands (disk destruction, privilege escalation, destructive git operations) for all subagents. When a specialist determines that a destructive operation is required, it does not attempt the operation; it returns to `tech-lead` with a structured request naming the exact command and the justification. `tech-lead` performs the operation from the main session, which operates in bypass mode and is not subject to the `deny` list. This centralizes all destructive shell execution at the single supervised orchestrator. The `deny` list is a coarse backstop — compound commands, piped invocations, and reversed-argument forms can evade prefix-glob matching — so the duty statement is the primary control and the `deny` list is defense-in-depth.

**Proposed addition to `.claude/agents/tech-lead.md`:**

> **Destructive Bash duty.** When a specialist returns a request for a destructive Bash operation (any command matching the deny list in `.claude/settings.json`, or any operation the specialist has flagged as irreversible), tech-lead evaluates the request, confirms the justification is sound, and executes the command from the main session. Tech-lead does not delegate destructive Bash operations back to a subagent. The main session's bypass mode is not a license for unsupervised destruction — tech-lead must review each request individually before executing.

### 9.5 Honest limitation of the deny list (for fw-adr-0030 ADR)

The `deny` list covers eighteen patterns across five categories: privilege escalation (`sudo`), storage destruction (`dd`, `mkfs`, `shred`, `truncate`, `rm`, `chmod 777`), destructive local git operations (`git reset --hard`, `git clean -f`, `git checkout --`, `git restore --`, `git branch -D`), and destructive remote git operations (`git push --force`, `git push --force-with-lease`, `git push -f`, `git push --delete`, `git push --mirror`). All eighteen are blocked at the permission layer before execution. However, Claude Code's `permissions.deny` mechanism is a coarse prefix-glob matcher that operates on the command string as presented — it does not parse shell syntax. The following evasion classes are not caught: compound commands where the destructive verb is not the leading token (e.g., `cd /tmp && rm -rf .`), bash-wrapped invocations (`bash -c "rm -rf /important"`), subshell forms (`$(git push --force ...)`), and reversed-argument ordering where the deny glob fails to match (e.g., `git push origin main --force` evades `Bash(git push --force*)`). The deny list is therefore a first-line catch for the most naive and accidental forms of these operations, not a containment boundary. The primary control is the duty statement in Hard Rule #13 (§9.4 above): the architectural constraint that destructive operations are categorically routed through tech-lead rather than executed by subagents. The deny list provides defense-in-depth against confused subagent behavior; the Hard Rule provides the governance constraint that holds even when the deny list does not match. Projects that require stronger containment should supplement with hook-based command inspection (a PreToolUse Bash hook that parses the command string with shell-grammar awareness) and remote-side protections (GitHub branch protection rules, required pull request reviews, tag protection rules) that operate independently of the permission posture.
