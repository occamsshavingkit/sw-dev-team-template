The v1.0.0-rc track was withdrawn after v1.0.0-rc2 proved unstable
under downstream use. **Returning to the rc track requires this
checklist to pass in full** — every row green, every supporting
artefact in place. The customer ratifies; `tech-lead` enforces;
`code-reviewer` audits at release-candidate time.

This file is the binding criteria. ROADMAP.md describes the path;
this file decides whether we have arrived. Owned by `tech-lead`
with `architect` concurrence on technical rows. Drafted in v0.15.0,
applied at the v1.0.0-rc3 cut.

---

## Status

- **Drafted:** 2026-04-25 (v0.15.0 prep).
- **Ratified by customer:** *pending — required before any v1.0.0-rc
  tag.*
- **First re-entry attempt target:** v1.0.0-rc3 (per ROADMAP.md).

## Criteria

Each row is binary: pass / fail. Half-credit not allowed. A failing
row blocks the rc cut.

### C-1 — Contract stability

**No open contract-breaking themes for at least one full MINOR
cycle** before the rc cut.

- *"Contract" =* the binding-rule surface in `CLAUDE.md` § Hard
  rules, the canonical-role roster, the agent contracts in
  `.claude/agents/*.md`, the upgrade contract in
  `scripts/upgrade.sh` + `migrations/*.sh`, and the SemVer
  semantics in `CHANGELOG.md`.
- *"Contract-breaking theme" =* an open issue or PR-bound proposal
  that, if landed, would force downstream projects to migrate
  beyond what `migrations/*.sh` can automate.
- *"One full MINOR cycle" =* the calendar duration between the most
  recent MINOR release and the next planned one — typically
  ~1 month.

**Pass condition:** the open-issue tracker shows zero items labelled
`contract-break` (or unlabelled but materially equivalent) older
than the last MINOR release.

**Status:** *pending*. Issues to triage at v0.15.0 close: TBD.

### C-2 — Migration infrastructure proven

**`migrations/*.sh` has been exercised across at least two major
hops with downstream projects, with zero customer escalations.**

- v0.11.0 → v0.12.0 hop: passed (lookback period: v0.12.0 release
  through v0.13.0 release, no escalations).
- v0.13.0 → v0.14.0 / v0.14.x hop: under test as of v0.14.3.
  Gating: zero escalations through the v0.15.0 release window.
- The atomic-install fix in v0.14.3 (issue #63) is part of the
  proof: a real downstream project surfaced the bug, the fix
  shipped, recovery procedure documented in CHANGELOG.

**Pass condition:** at v0.15.0 close, no open issues labelled
`migration-defect` traceable to either hop.

**Status:** *pending — v0.14.x post-release window in progress.*

### C-3 — Retrofit playbook field-tested

**The retrofit playbook (`docs/templates/retrofit-playbook-template.md`,
v0.13.0+) has been used against at least one real existing
codebase outside the template's own examples directory, with the
retrofit DoD met.**

- *"Real existing codebase" =* a project with prior independent git
  history, brought into a scaffolded template via the playbook.
- *"DoD met" =* the playbook's exit checklist signed off; the
  retrofitted project's first post-retrofit milestone closed
  cleanly.

**Pass condition:** at least one downstream attestation
(`docs/audits/<project>-retrofit-summary.md` in that project's
own repo, with link recorded back upstream via issue-feedback).

**Status:** *pending — currently zero retrofits attempted.*

### C-4 — Workflow-pipeline empirical usage

**All four stages of the workflow-redesign pipeline (prior-art /
three-path / proposal / Solution Duel) have empirical usage data
from real downstream tasks, with at least one per-stage tuning
pass applied based on that data.**

- Prior-art (`docs/prior-art/<task-id>.md`): at least 5 real-task
  applications across ≥ 2 downstream projects.
- Three-path ADRs: at least 5 ADRs ratified through the M/S/C
  shape; FW-ADR-0001 through FW-ADR-0007 in this repo count toward this
  bar (7 already, threshold met).
- Proposal stage (`docs/proposals/<task-id>.md`): at least 3
  proposals reviewed in real tasks.
- Solution Duel: at least 3 duels held; outcome data recorded.

**Pass condition:** evidence in `docs/pm/LESSONS.md` (or per-
project equivalent); per-stage tuning recorded in the same.

**Status:** *partial. Three-path: green (7 ADRs in this repo
already, plus the customer's downstream projects). Other three:
pending downstream evidence.*

### C-5 — Audit agents exercised

**`onboarding-auditor` and `process-auditor` have each run at least
twice against the template's own examples (or downstream
projects), with no outstanding major findings.**

- `onboarding-auditor` zero-context audit: produces
  `docs/pm/FRICTION_REPORT-<date>.md`. Major findings escalated
  to `tech-writer` / `architect` / `researcher` /
  `project-manager` per the friction-report contract.
- `process-auditor` ritual-debt audit: produces a
  `process-audit-<date>.md` report. Major findings either
  ratified by customer (rituals retained), retired, or argued
  for retention with explicit rationale recorded.

**Pass condition:** at v0.15.0 close, no outstanding **major**
findings from either auditor against the template repo or
representative downstream.

**Status:** *pending — auditors exist (added in v0.12.0); usage
data on this template repo TBD.*

### C-6 — `v2-proposal` queue cleared

**Every issue currently labelled `v2-proposal` (or materially
equivalent — items proposed for the next major) is either
landed in v0.x, formally deferred to v2.0 with a
`docs/v2/<topic>.md` placeholder, or explicitly rejected with
reason in `docs/DECISIONS.md`.**

- "Landed" = in a tagged release.
- "Deferred to v2.0" = `docs/v2/<topic>.md` exists with a one-
  paragraph "what" + "why-not-now" + "what-blocks" sketch.
- "Rejected" = a `D-NNNN` row in `DECISIONS.md`.

**Pass condition:** zero issues labelled `v2-proposal` in an
ambiguous state at v0.15.0 close.

**Status:** *pending — label inventory required at v0.15.0 close.*

### C-7 — End-to-end upgrade smoke

**`scripts/upgrade.sh` upgrades a synthetic v0.14.4 project
through every intermediate release to the v1.0.0-rc3 candidate
cleanly, with `--verify` exit 0 at each hop.**

- *"Synthetic v0.14.4 project" =* a freshly-scaffolded project
  hand-stamped to v0.14.4 with no customisations. The original
  draft of this row read "v0.10.0"; re-baselined to v0.14.4 on
  2026-04-25 because v0.13.0–v0.14.2's in-place `cp` pattern
  mutates the running upgrade.sh's inode mid-execution. Projects
  on those versions need the one-time curl recovery documented
  in v0.14.3's CHANGELOG to land v0.14.4's upgrade.sh; from there
  they are stepwise-clean. The harness still accepts `--start`
  for testing pre-v0.14.4 hops as historical-regression
  documentation, but the rc3 pass condition is the supported
  range.
- *"Cleanly" =* upgrade.sh exits 0 at each hop, no manual
  intervention, no stale `.tmp.*` files.
- The smoke-test simulation already covers v0.1.0 →
  vCURRENT in one step (per `scripts/smoke-test.sh`); the rc3
  smoke must also cover the **stepwise** path (one hop at a
  time) to catch any cross-version regression in
  `migrations/*.sh`.
- The harness uses git's `insteadOf` rewrite (via
  `GIT_CONFIG_COUNT/KEY/VALUE` env) to redirect the GitHub
  upstream URL to a local pinned clone. This lets each hop run
  its OWN historical upgrade.sh naturally — bootstrap,
  re-exec, and all — exercising the real upgrade flow rather
  than a HEAD-pre-installed surrogate.

**Pass condition:** a stepwise smoke run, captured as
`docs/audits/v1.0.0-rc3-stepwise-smoke.log`, shows clean exits
through every release between v0.14.4 and the rc3 candidate.

**Status:** *green for the supported range as of 2026-04-25 PM
session — 4/4 hops pass (v0.15.0 → v0.17.0). Re-capture log at
the rc3 cut so it covers v0.14.4 → rc3 inclusive.*

## Audit + sign-off

When the customer believes all rows pass:

1. `code-reviewer` runs an IEEE 1028-style audit pass over the
   checklist and supporting artefacts, producing
   `docs/audits/v1.0.0-rc3-readiness-audit.md`.
2. `tech-lead` presents the audit summary to the customer.
3. Customer ratifies (or returns specific rows for more work).
4. On ratification, `release-engineer` cuts v1.0.0-rc3.

Until step 4, the v0.y track continues. **No premature rc tag.**
