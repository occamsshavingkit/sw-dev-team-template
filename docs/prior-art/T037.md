## 3. Canonical solution

- **Canonical pattern found:** yes.

The schema should validate the repository's provider-neutral routing record shape already defined in `docs/model-routing-guidelines.md`: `tier`, `effort`, `model`, fallback event fields, release verification dates, and authority-preservation assertions. JSON Schema 2020-12 is the right validation language; exact provider model IDs remain release-verified evidence, not schema constants except for internal enumerations such as tier and effort.

## 5. Known pitfalls

- Hard-coding stale provider IDs: local policy requires exact IDs and aliases to be verified at release time, so schema should permit provider/model strings while requiring `verified_on` where exact IDs are pinned.
- Fallback under-validation: fallback records need controlled reason values and an explicit authority-preservation field, otherwise weaker-provider fallbacks can silently alter governance.
- Confusing effort with authority: schema can validate `effort`, but policy still decides role authority and review gates.

## 6. Recommendation to `architect` / `software-engineer`

Define a small JSON Schema 2020-12 model for routing entries and fallback events that mirrors `docs/model-routing-guidelines.md`, using enums only for stable policy values (`fast`, `standard`, `strong`, `frontier`; `low`, `medium`, `high`, `xhigh`, `max`; approved fallback reasons) and requiring release-verification evidence when exact model IDs are recorded.
