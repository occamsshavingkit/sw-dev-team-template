## 3. Canonical solution

- **Canonical pattern found:** yes.

GitHub Actions supports manual `workflow_dispatch` inputs and scheduled runs, and pull requests are GitHub's collaboration and review boundary for proposed changes. GitHub's security guidance warns that allowing workflows to create or approve PRs can be risky without oversight, so this workflow must be gated, least-privileged, and unable to merge or approve its own output.

## 5. Known pitfalls

- `workflow_dispatch` inputs are capped by GitHub limits; do not encode broad task plans into one payload.
- Scheduled workflows run on default branch and can create surprise PRs unless explicitly gated.
- Workflows that can create PRs must not also approve or merge them.
- Failure handling should create a no-op status or issue output, not a broken commit or half-created PR.

## 6. Recommendation to `architect` / `software-engineer`

Prefer `workflow_dispatch` first, with optional low-frequency schedule only after review; require one issue/gap input, run contract checks before PR creation, create a branch/PR only after checks pass, and report no-op or issue output on failure.
