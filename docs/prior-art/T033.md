## 5. Known pitfalls

- Missing timestamps: without an ISO-8601 timestamp, release review cannot tie fallback to a run.
- Missing task IDs: fallback evidence must connect to the task or dispatch that used it.
- Authority drift: fallback records must say the original role authority and output format still apply.
- Secret leakage: fallback reasons should be categorical (`credit_exhausted`, `provider_unavailable`, `policy_blocked`, etc.), not copied provider error payloads that may include sensitive data.

## 6. Recommendation to `architect` / `software-engineer`

Add a required fallback log shape to `docs/model-routing-guidelines.md` using the source-plan fields plus a short authority-preservation rule: changing the model provider does not change role ownership, output format, review obligations, or canonical-source hierarchy.
