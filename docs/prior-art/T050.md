## 3. Canonical solution

- **Canonical pattern found:** partial.

GitHub Actions provides workflow steps, shell commands, and PR path/diff context that can enforce a repository-defined patch-size policy, but GitHub does not define a universal maximum changed-file or diff-line threshold. The threshold is a project policy decision that should be stated in `docs/workflow-pipeline.md` and implemented by `improve-template.yml`.

## 5. Known pitfalls

- GitHub path filters have documented diff-size limits; enforcement should inspect the checked-out branch diff directly rather than rely only on trigger filters.
- Counting generated or vendored artifacts can distort patch size unless the policy says whether they are included.
- Oversized output must block automation before PR creation or update the PR with a clear failure, depending on when the size is known.

## 6. Recommendation to `architect` / `software-engineer`

Define explicit numeric limits in `docs/workflow-pipeline.md`, then implement the same constants in `improve-template.yml`; block automation output that exceeds either changed-file count or diff-line count and emit a no-op or issue update with the measured totals.
