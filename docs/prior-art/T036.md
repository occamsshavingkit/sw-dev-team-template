## 3. Canonical solution

- **Canonical pattern found:** yes.

Use JSON Schema 2020-12 as the contract language, with `$schema`, `$id`, explicit required fields, closed object shapes where appropriate, and schema self-validation before applying the schema to agent files. JSON Schema is the canonical validation vocabulary; Ajv is an implementation candidate, not the contract authority. The schema should model canonical agent frontmatter/body invariants already present in `.claude/agents/*.md`, not invent a new role system.

## 5. Known pitfalls

- Draft mismatch: Ajv CLI defaults to draft-07 unless `--spec=draft2020` is supplied; downstream validation should pin the draft explicitly.
- Schema overreach: validation must preserve role authority from canonical Markdown and must not turn generated/runtime artifacts into policy sources.
- YAML/frontmatter ambiguity: agent files are Markdown with YAML frontmatter; schema validation likely needs a deterministic extraction step before JSON validation.

## 6. Recommendation to `architect` / `software-engineer`

Use JSON Schema 2020-12 for `agent-contract.schema.json`, validate the schema itself, and keep the data model limited to observable contract fields from canonical role files: role name, description, tools, model, local-supplement rule, escalation/customer-interface constraints, output shape, and workflow-pipeline hooks.
