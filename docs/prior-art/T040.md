## 3. Canonical solution

- **Canonical pattern found:** partial.

Canonical local pattern: generated runtime artifacts must be reproducible, subordinate to canonical inputs, and reviewed before operational use. External prior art supports token counting and Markdown-to-context compilation, but there is no single standard for compiling multi-agent role Markdown into provider-specific runtime prompts. The safe pattern is deterministic compilation plus line counts always, token counts when provider-specific counters are available, and explicit fallback to line/byte reporting when they are not.

## 5. Known pitfalls

- Token counts are provider-specific estimates: Anthropic docs warn actual message tokens can differ slightly and include system optimizations; tiktoken applies to OpenAI encodings only.
- Hidden nondeterminism: generated outputs need stable ordering, canonical input list, generator metadata, and checksum/freshness evidence.
- Markdown ambiguity: CommonMark gives parsing rules, but agent contract semantics still come from canonical project files.

## 6. Recommendation to `architect` / `software-engineer`

Build `compile-runtime-agents.sh` as a deterministic generator that reads canonical inputs, writes generated runtime candidates with provenance, reports line counts unconditionally, reports provider-specific token estimates only when configured tools/API credentials are available, and fails drift/freshness checks without making generated files canonical.
