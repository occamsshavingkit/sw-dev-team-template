## 3. Canonical solution

- **Canonical pattern found:** yes.

Use JSON Schema to validate generated-artifact manifests rather than generated prose directly. The manifest should identify artifact class, generated status, canonical inputs, generator command or version, output path, checksum/freshness data, review gate, and retention. This matches local prior art in `docs/runtime/agents/README.md`, where runtime candidates are explicitly generated and subordinate to canonical inputs.

## 5. Known pitfalls

- Validating generated content instead of provenance: the schema should prove reproducibility metadata and authority boundaries, not try to encode every Markdown semantic.
- Manual drift: generated outputs need checksums or source/generator timestamps so hand edits fail freshness checks.
- Authority inversion: generated artifacts must cite canonical inputs and review gates so consumers do not treat generated candidates as policy.

## 6. Recommendation to `architect` / `software-engineer`

Model a generated-artifact manifest with required fields for `artifact_type`, `authority`, `canonical_inputs`, `generator`, `generated_at`, `output_path`, `checksum`, `review_gate`, and `retention`; use it to validate runtime candidates and future session-start/adapter outputs before M6 acceptance.
