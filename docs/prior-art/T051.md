## 3. Canonical solution

- **Canonical pattern found:** yes.

GitHub pull request reviews support comment, approval, and request-changes states, and branch protection can require approving reviews and status checks before merge. The project already has Hard Rule #3, "No commit without `code-reviewer` review," so M7 automation PRs need a documented human sequence that automation cannot satisfy by approving itself.

## 5. Known pitfalls

- Branch protection settings are repository configuration, not fully captured by committed docs unless the repository also uses rulesets or policy-as-code.
- Required reviews can become stale after new commits; the workflow must not push after final human approval without re-review.
- Reviewer role sequence must include `code-reviewer`; release/security roles are also relevant for workflow and automation changes.

## 6. Recommendation to `architect` / `software-engineer`

Document the M7 review sequence as implementation owner review, `release-engineer` workflow review, `security-engineer` automation/security review where workflow permissions or bot behavior are touched, and final `code-reviewer` approval before merge; require status checks and prohibit automation from approving or merging its own PR.
