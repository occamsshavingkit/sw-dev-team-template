## Final Decision Rule

`v1.0.0` is `final-ready` only when every gate above is green or has an
explicit customer-approved exception. Exceptions must name the waived
gate, rationale, approver, date, and residual risk in this file and in
`docs/pm/CHANGES.md`.

## GitHub Release Object Steps

1. Confirm the final candidate commit is the reviewed commit set.
2. Create the annotated `v1.0.0` tag from that commit.
3. Push the tag after release-engineer approval.
4. Create the GitHub Release object for `v1.0.0`.
5. Include release notes with issue closure links for #71-#105, upgrade
   notes, rollback notes, and downstream compatibility guidance.
6. Verify the Release object points to the intended tag and that issue
   references / closure comments are complete.

## Rollback / Upgrade Note Requirements

- Identify the previous known-good tag and the `v1.0.0-rc6` candidate
  as rollback targets until G-1 downstream clean-window / sample
  evidence is recorded as green.
- Tell downstream projects to run `scripts/upgrade.sh --dry-run`
  before applying the upgrade.
- Call out volatile-file and `.claude/agents/*-local.md` supplement
  behavior so project-specific routing survives upgrades.
- Note that issue #84 does not rewrite `v1.0.0-rc3`; if a `--dry-run`
  unexpectedly performed the upgrade in an rc3-era downstream tree,
  inspect the worktree diff, keep and commit only after review or
  restore from VCS, then use current rc6 `scripts/upgrade.sh --dry-run`
  from a clean branch/worktree for later previews.
- Require rerunning smoke tests and the issue-feedback onboarding check
  after upgrade.
