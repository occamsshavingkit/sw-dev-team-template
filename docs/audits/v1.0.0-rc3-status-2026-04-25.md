## Findings (2026-04-25 PM session)

### C-7 — bug found and fixed in `scripts/stepwise-smoke.sh`

The first run of the harness (commit baseline) failed at hop v0.15.1
with `TEMPLATE_VERSION expected v0.15.1, got v0.17.0`. Root cause
diagnosed across four iterations:

- `SWDT_UPSTREAM_URL` env-var support was added in v0.16.0; pre-v0.16.0
  upgrade.sh hardcodes the GitHub URL.
- The harness pinned `$clone`'s `main` to the per-hop tag (correct),
  but each hop's local upgrade.sh had been **bootstrap-cp'd from the
  hop's $clone**, downgrading it to a hardcoded-URL version. The next
  hop then cloned GitHub instead of `$clone`, landing at HEAD's tag
  (v0.17.0) regardless of `--target`.
- Symptom: hop 1 (v0.14.4 → v0.15.0) accidentally bootstrap-cp'd
  GitHub's HEAD upgrade.sh (which has `SWDT_UPSTREAM_URL` support)
  → looked like it worked. Hop 2 then bootstrap-cp'd v0.15.1's
  upgrade.sh (no `SWDT_UPSTREAM_URL`) → broke.

**First fix attempt (workaround, superseded):** pre-install HEAD's
`scripts/upgrade.sh` and `scripts/lib/manifest.sh` into the synthetic
project at scaffold time, then run with `SWDT_BOOTSTRAPPED=1` to
suppress the bootstrap self-update path, plus `--target $tag` per hop.
This worked (4/4 green) but exercised a HEAD-pre-installed surrogate,
not the real upgrade flow.

**Root-cause fix landed (PM session, this date):** harness now uses
git's `insteadOf` rewrite via `GIT_CONFIG_COUNT/KEY/VALUE` env to
redirect the GitHub upstream URL to the local pinned clone. With
the redirect in place, each hop runs its OWN historical upgrade.sh
naturally — bootstrap, re-exec, and all — and clones still hit the
pinned local repo. The pre-install / `SWDT_BOOTSTRAPPED` /
`SWDT_UPSTREAM_URL` / `--target` workaround is removed; tag-masking
remains (it controls which tag the historical "latest tag" picker
sees, since pre-v0.17.0 upgrade.sh has no `--target` flag).

Canonical run log: `docs/audits/v1.0.0-rc3-stepwise-smoke.log`
(re-captured after root-cause fix).
Result: `4/4 hops passed (v0.15.0 → v0.17.0)`.

**Checklist amendment landed (this date).** C-7 row re-baselined from
"v0.10.0 → rc3" to "v0.14.4 → rc3" with revision-log entry. Pre-v0.14.4
upgrade.sh has a documented self-mutating-cp regression no harness fix
can paper over (those projects need the v0.14.3 CHANGELOG curl
recovery). The harness still accepts `--start` for historical-
regression documentation; the binding pass condition is the supported
range.
