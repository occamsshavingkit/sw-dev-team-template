## Context and problem statement

From template v1.0 through v1.5.2 the scaffold set
`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` in `.claude/settings.json` and
relied on the persistent named-teammate model it enables: `tech-lead`
spawned specialists with a `name` parameter, teammates appeared on the TUI
status panel, and mid-task escalation flowed push-style via `SendMessage`.

Two upstream Claude Code bugs surfaced during v1.5.3 work:

- **Bug #355:** When the main session is driven under remote control (MCP
  tool-bridge invocation, headless CI), permission prompts generated by
  spawned subagents are not surfaced to the operator — they silently time out
  or block. The flag is the precondition: the prompt-routing failure only
  occurs when the experimental feature is active.
- **Bug #356:** Subagents spawned under the experimental flag do not inherit
  the main session's permission mode. Even when the project sets
  `permissions.defaultMode: acceptEdits` in `settings.json`, a spawned
  subagent reverts to interactive-prompt mode. This directly caused the
  Hard Rule #1 violation observed in v1.5.2 (a `librarian` subagent's write
  to `CUSTOMER_NOTES.md` surfaced a prompt to the human terminal). The
  elaborate `customer-notes-guard.py` role-aware fix designed in the
  permission-leak design doc was the proposed workaround — but the root
  cause is the flag, not the hook.

A targeted test confirmed the diagnosis: with the flag removed and
`permissions.defaultMode: acceptEdits` set in `settings.json`, a subagent
inherited the permission mode and wrote silently without a prompt. The flag
is the single point of failure. Both bugs are filed upstream and tracked,
but no fix timeline is available.

The ADR trigger is a cross-cutting pattern change: the spawning model,
escalation protocol, all 16 agent `tools:` declarations, and the settings
surface all change simultaneously.

---

## Decision drivers

- **Bug #356 is the direct cause of a Hard Rule #1 violation.** Permission
  prompts from spawned subagents route to the human terminal — only
  `tech-lead` may interface with the customer. This is not a theoretical
  risk; it was observed in production in v1.5.2.
- **Bug #355 breaks remote-control workflows silently.** Headless invocations
  (CI, MCP) block without surfacing the prompt. Operators cannot distinguish
  a hung workflow from a slow one.
- **Both bugs are flag-specific.** Removing the flag eliminates both failure
  modes with a single settings change. No workaround within the flag-on world
  achieves the same guarantees without significant hook complexity.
- **The one-shot spawning model is the standard Claude Code pattern.** It
  does not depend on experimental feature availability, survives harness
  version upgrades, and is consistent with the Codex and Antigravity adapter
  models already in use.
- **Lost capabilities are recoverable operationally.** Mid-task push
  escalation (`SendMessage`) and liveness pings are replaced by pull
  escalation (blocker in return value) and file-based health signals —
  patterns already used by the Codex and Antigravity paths.
- **The customer intends to re-evaluate.** Bugs #355 and #356 are tracked
  upstream. The decision is reversible once they are fixed. An explicit
  revisit trigger is recorded below.

## Considered options (Three-Path Rule, binding)

### Option M — Minimalist: document the bugs, keep the flag, add workarounds

Accept the experimental flag as-is and mitigate Bug #356 via an expanded
hook layer: implement the role-aware `customer-notes-guard.py` redesign
(shared `role_resolver.py` module, `librarian` → allow, other specialists →
deny, main-session → ask) as specified in
`docs/pm/v1.5.3-hr1-permission-leak-design.md`. Mitigate Bug #355 by
documenting that remote-control / headless invocations are unsupported while
the flag is on.

- **Sketch:** Ship the role-aware hook layer as designed. Add a
  `settings.json` `permissions.defaultMode: acceptEdits` line as the
  defensive floor. Document bug #355 as a known limitation. No changes to
  agent files, spawning model, or escalation protocol.
- **Pros:**
  - Named-teammate persistence and `SendMessage` mid-task push escalation are
    retained.
  - The TUI status panel shows active specialists.
  - The hook-based fix is surgical — only two scripts and one new module
    change.
- **Cons:**
  - Bug #356 is a harness-level failure. The hook can gate write decisions
    after the fact, but it cannot prevent the harness from routing a prompt to
    the customer if any other hook or tool invocation triggers an `"ask"`.
    The fix is incomplete by construction.
  - Bug #355 remains unmitigated for headless use cases. Any spawn under
    remote control that generates a prompt silently blocks.
  - The `agent_type` field in hook payloads is not a guaranteed stable
    harness contract — it is an experimental-feature detail that could change
    without notice in a future Claude Code version.
  - The role-aware hook adds ~300 lines of hook code and a new shared module
    to maintain, all to paper over a flag-specific failure. The complexity
    sits in the scaffolding permanently even after the bugs are fixed
    upstream.
- **When M wins:** if named-teammate persistence is a hard product
  requirement, or if upstream bug fixes are confirmed to land within the
  current release cycle. Neither holds.

### Option S — Scalable: remove the flag, revert to one-shot subagents, reshape escalation model

Remove `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` from `.claude/settings.json`.
Add `permissions.defaultMode: acceptEdits` as the harness floor. Update all
16 agent `tools:` lines and bodies to remove `SendMessage` references. Rewrite
the escalation model from push (mid-task `SendMessage`) to pull (blocker
embedded in the specialist's return value, `tech-lead` acts on it). Replace
persistent named teammates with respawn-as-re-dispatch. Update
`SW_DEV_ROLE_TAXONOMY.md`, scaffold/respawn/audit scripts, agent-contract
schema, the bundled example, and all generated runtime mirrors.

- **Sketch:** Two-phase change. Phase 1: settings.json flag removal +
  `defaultMode` addition. Phase 2: agent files, scripts, schema, and doc
  regeneration. The escalation shape shifts from `SendMessage({to: name})`
  to a structured `BLOCKER:` section in the specialist's return text;
  `tech-lead` reads it and dispatches the next wave. File-based health
  signals (e.g., writing to a temp status file) replace liveness pings.
  Respawning a specialist is re-dispatching it with amended context —
  same `Agent` tool call, new brief.
- **Pros:**
  - Eliminates both bugs at the root: no flag, no flag-specific failures.
  - `permissions.defaultMode: acceptEdits` now takes effect for all subagents
    without exception, removing the need for the role-aware hook redesign.
  - The one-shot model is the stable Claude Code contract. It is harness-
    version agnostic and consistent with Codex and Antigravity adapter
    behavior.
  - The pull-escalation model is already the pattern used by Codex and
    Antigravity sessions; unifying on it removes a fork in the orchestration
    logic.
  - No new maintenance surface: the hook complexity from Option M is not
    needed.
  - The decision is fully reversible: re-enabling the flag and restoring
    agent files is a targeted change once upstream fixes land.
- **Cons:**
  - Mid-task push escalation is lost. A specialist that hits a mid-task
    blocker cannot interrupt `tech-lead` immediately; the blocker surfaces
    only when the specialist returns. For tasks with long execution times
    this delays blocker detection by one dispatch cycle.
  - Named-teammate persistence across turns is lost. Each dispatch is a fresh
    spawn; in-progress state must be handed through the brief, not through a
    persistent channel.
  - Liveness pings are lost. There is no in-band signal that a specialist is
    still running (file-based workaround is possible but is out-of-band).
  - The migration scope is broad: 16 agent files, settings.json, scripts,
    schema, CLAUDE.md, AGENTS.md, GEMINI.md, `.agents/rules/`, and generated
    runtime mirrors all require coordinated updates.
- **When S wins:** the flag-specific bugs are the proximate cause of a Hard
  Rule violation and a silent remote-control failure, the upstream fix
  timeline is unknown, and the lost capabilities can be operationally
  approximated without the flag. All three hold.

### Option C — Creative: selective flag use gated on harness-version detection

Keep the flag as the default but add a harness-version detection layer:
at session start, query the Claude Code version via a shell hook or env
variable, compare it against a `KNOWN_GOOD_VERSIONS` list maintained in
`scripts/`, and conditionally suppress the flag if the version is in the
known-buggy set. Agents operate in flag-on mode when safe and flag-off mode
when not.

- **Sketch:** A `PreSessionStart` or `SessionInit` hook reads
  `CLAUDE_CODE_VERSION` (or a similar env variable), checks it against a
  maintained allowlist, and either no-ops (flag active) or temporarily
  removes the flag key from the settings payload (flag inactive). Agents
  carry `SendMessage` in their `tools:` lines with a conditional instruction:
  "use `SendMessage` only if `AGENT_TEAMS_ACTIVE` is set in environment".
  The dual-mode agent body adds an `if/else` branch to each escalation
  pattern.
- **Pros:**
  - Named-teammate persistence is available when the harness version is
    known-good.
  - The `KNOWN_GOOD_VERSIONS` list can be updated without an ADR, making
    re-enablement low-friction once upstream fixes land.
- **Cons:**
  - `CLAUDE_CODE_VERSION` availability in hook context is not confirmed from
    a Tier-1 source. If the variable is absent, the detection logic fails
    silently and the flag remains on — reproducing Bug #356 in the worst case.
  - Dual-mode agents (SendMessage + pull-escalation branches) are materially
    more complex than either single-mode option. All 16 agent files carry dead
    code for the inactive mode.
  - The `KNOWN_GOOD_VERSIONS` list requires active maintenance. A Claude Code
    update that does not announce changes to the experimental feature could
    silently re-introduce the bug on a version not yet in the deny list.
  - Testing both modes in CI requires a harness-version fixture that does not
    currently exist and would be difficult to maintain hermetically.
  - This is precisely the kind of "workaround complexity" that Option S
    eliminates. The constraint that rejects Option C: the harness detection
    surface is unconfirmed and dual-mode agent bodies are a long-term
    maintenance liability.
- **When C wins:** the experimental feature provides capabilities that cannot
  be approximated without it (e.g., real-time cross-agent state sharing that
  pull escalation cannot model), the version detection surface is confirmed
  stable, and the team has bandwidth for dual-mode maintenance. None of these
  hold here.

---

## Decision outcome

**Chosen option: S — remove the flag, revert to one-shot subagents, reshape
escalation model.**

Option M is incomplete by construction: it papers over a harness-level bug
with hook complexity and leaves Bug #355 unmitigated. Option C introduces
unconfirmed harness detection surface and permanent dual-mode agent
complexity. Option S removes the failure mode at the root, is consistent with
the Codex and Antigravity adapter models already in production, and is fully
reversible once upstream fixes land. The lost capabilities (mid-task push
escalation, persistent teammates, liveness pings) are operationally
approximatable via pull escalation and file-based signals — a trade-off the
customer accepted given the alternative of a live Hard Rule #1 violation.

## Consequences

### Positive

- Hard Rule #1 violation is eliminated at the root: subagents inherit
  `acceptEdits` and do not route permission prompts to the customer terminal.
- Headless and remote-control invocations (CI, MCP tool-bridge) are no longer
  susceptible to silent blocking from unrouted subagent prompts.
- Agent files are simpler: no `SendMessage` tool, no dual-escalation branches,
  no TUI panel references.
- The hook layer required for Option M is not needed, removing ~300 lines of
  hook complexity and the `role_resolver.py` shared module.
- Consistent spawning model across all four supported harnesses (Claude Code,
  Codex, Gemini CLI, Antigravity).
- The decision is fully reversible: re-enabling the flag is a targeted
  `settings.json` edit and agent-file restore, bounded to the artifacts listed
  in the migration scope above.

### Negative / trade-offs accepted

- **Mid-task push escalation lost.** A specialist hitting a blocker mid-task
  cannot interrupt `tech-lead` until it returns. Blocker detection is delayed
  by one dispatch cycle. Mitigation: specialists embed a structured `BLOCKER:`
  section at the top of their return output; `tech-lead` checks for it before
  acting on the rest of the output.
- **Named-teammate persistence lost.** Each specialist dispatch is a fresh
  spawn; in-progress state must be passed through the dispatch brief. Multi-
  turn specialist work that previously accumulated shared state across turns
  must now be checkpointed and re-supplied in each brief.
- **Liveness pings lost.** There is no in-band signal that a specialist is
  still running. File-based health signals (writing a sentinel file to a
  known path at dispatch start and clearing it on return) provide a partial
  out-of-band substitute; this is noted as an operational gap accepted under
  this decision.
- **TUI status panel no longer shows active specialists.** This is a developer-
  experience regression with no operational substitute in the one-shot model.

## Revisit trigger (binding)

The customer has stated the intent to monitor upstream Claude Code bugs #355
and #356 and re-evaluate re-enabling `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`
once both are confirmed fixed. This is an explicit revisit trigger:

**When bugs #355 and #356 are both marked resolved in the upstream Claude
Code issue tracker, the first session after that confirmation must open a
superseding ADR under the Three-Path Rule.** That ADR must document:

1. The confirmed fix for each bug (link to upstream resolution).
2. A verification step: re-run the test that originally confirmed the
   diagnosis (subagent inherits permission mode and writes silently) — this
   time confirming that the flag-on path also passes.
3. The migration scope to re-enable: the inverse of the table in the
   "Migration scope" section above.

Until both bugs are confirmed resolved, this ADR stands and the flag remains
off. Partial resolution (one bug fixed, one open) does not trigger the
revisit; both must be fixed.

## Verification

- **Success signal:** A subagent spawned via the `Agent` tool (no flag)
  inherits `permissions.defaultMode: acceptEdits` from the project
  `settings.json` and completes a write operation without routing a prompt to
  the operator terminal. Confirmed in test (2026-06-16).
- **Failure signal (regression):** Any subagent write, bash execution, or
  tool call surfaces a permission prompt to the human terminal. This indicates
  either `permissions.defaultMode` is not set, a hook is still returning
  `"ask"` unconditionally, or a harness update has changed permission
  inheritance. Open a bug and escalate immediately.
- **Failure signal (revisit):** Upstream Claude Code issues #355 and #356
  are marked resolved. Triggers the superseding ADR process described in
  the Revisit trigger section.
- **Review cadence:** First session of each calendar quarter, check the
  upstream status of bugs #355 and #356. If either is resolved, queue the
  superseding ADR.
