# v1.0.0-rc4 Stabilization Plan

Field-use triage for issues filed after `v1.0.0-rc3`. This plan
decides what must land before `v1.0.0` final, what belongs in the
integrated `v1.0.0-rc4` candidate, and what can move to post-GA work.

Status: **rc4 tagged / rc5 release-prep / not final-ready**, opened
2026-05-03 after downstream use of `v1.0.0-rc3` surfaced multiple
framework gaps. Post-rc4 downstream validation surfaced issues #84
through #103; those are now the `v1.0.0-rc5` release-boundary set.

Release-state vocabulary used across the rc4 plan, review, and roadmap:

| State | Meaning |
|---|---|
| `draft` | Plan is still being shaped and has not completed specialist review. |
| `release-prep` | Release files are being updated for a candidate tag, but review and tagging are not complete. |
| `review-complete` | In-tree candidate work has passed the recorded specialist review and smoke evidence, but the release candidate is not tagged. |
| `tagged` | The annotated `v1.0.0-rc4` git tag exists on the reviewed commit. |
| `final-ready` | `docs/v1.0.0-final-checklist.md` is complete and every final gate is green. |

Current rc4 state is `tagged`, but not `final-ready`. Current rc5
state is `release-prep`; it must not be described as `tagged` or
`final-ready` until those objective gates are satisfied.

## Release decision

`v1.0.0-rc3` does not go directly to `v1.0.0` final. The project cuts
`v1.0.0-rc4` first, containing the release-blocking process,
scoping, upgrade, and retrofit fixes below. `v1.0.0` final waits for
the objective gates in `docs/v1.0.0-final-checklist.md`, including the
recorded downstream clean-window / sample gate, to go green or receive
explicit customer-approved exceptions. After rc4 was tagged,
downstream validation found issues #84 through #103, so the project
cuts `v1.0.0-rc5` before final.

## Severity classes

| Class | Meaning | Release handling |
|---|---|---|
| P0 | Blocks `v1.0.0` final because the current contract can mis-capture customer intent, bypass required specialist review, leak sensitive content, or leave scaffold / upgrade users in a known-bad state. | Must land in `v1.0.0-rc4`. |
| P1 | Important stability or operator-friction fix; should land in rc4 if small or tightly coupled to a P0, otherwise can follow in v1.0.x / v1.1.0. | Prefer rc4, but not final-blocking unless escalated. |
| P2 | Design or enhancement track; real issue, but not required for GA stability. | Post-GA or v2. |

## rc4 work packages

Status legend: `review-complete` means the template-local implementation
passes smoke and has agent-team review for rc4 continuation. GitHub
issue comments / closures still happen after the rc4 commit exists.

### WP-1 — First-session and upgrade gates

Issues: #73, #78.

Status: `review-complete`.

Deliverables:

- Session-start detection for missing FIRST ACTIONS / Step 0 record.
- Upgrade-time `ACTION REQUIRED` warning for projects without issue-
  feedback opt-in.
- `docs/ISSUE_FILING.md` note that retroactive opt-in is valid.
- Smoke coverage for fresh scaffold and upgraded-project warning paths.

### WP-2 — Tech-lead routing discipline

Issues: #77, #82.

Status: `review-complete`.

Deliverables:

- Add an explicit hard rule that `tech-lead` orchestrates and does not
  author production artifacts directly except for narrowly defined
  orchestration / tool-bridge cases.
- Strengthen `CUSTOMER_NOTES.md` ownership: customer-answer prose is
  written by `researcher`, not inline by `tech-lead`.
- Add a session self-audit or pre-commit / pre-close checklist that
  flags non-trivial main-session edits that should have been routed to
  a specialist.

### WP-3 — Subagent reliability and liveness

Issues: #72, #74.

Status: `review-complete`.

Deliverables:

- Rewrite liveness-ping language so it is truthful under current
  harness limits: `SendMessage` when available, otherwise progress
  journal / structured final status.
- Add dispatch-sizing guidance: split tasks with high source-document
  fanout or multiple outputs.
- Add completion verification: expected file changes, non-future-tense
  final summary, and retry path on zero-output returns.

### WP-4 — Scoping intent fidelity

Issue: #80.

Status: `review-complete`; new canonical role deferred pending customer
ruling.

Deliverables:

- Add a deliverable-shape question to Step 2 scoping.
- Add a glossary gate for customer-domain terms used in the deliverable
  definition.
- Add a review gate that checks output shape against the customer-
  ratified deliverable definition.
- Decide whether a new `skill-author` / `prompt-engineer` role is rc4
  scope or post-GA; avoid adding a new canonical role in rc4 unless
  the customer explicitly accepts the contract change.

### WP-5 — Retrofit safety and close-out

Issues: #81, #83.

Status: `review-complete`.

Deliverables:

- Require identifying-content regex sweeps in Stage A, implementer
  self-checks, code-reviewer audits, and security-engineer sign-off.
- Require per-hit / per-line tables; aggregate verdicts are not enough.
- Add source-remote propagation from Stage A into a close-out remote
  disposition decision.
- Add remote disposition to the retrofit closure artifact.

### WP-6 — Upgrade conflict policy and local agent supplements

Issues: #71, related #75.

Status: `review-complete`; conflict heat-map, volatile-file policy, and
convention-based `<role>-local.md` supplements implemented.

Deliverables:

- Document volatile-file policy for scripts and harness settings.
- Preserve append-only governance files without upstream reformat churn
  where possible.
- Add upgrade conflict output that helps decide whether to take
  upstream, preserve local, or merge.
- Add project-owned `<role>-local.md` agent supplements that canonical
  agents read before role work, while preserving upstream canonical
  agent files across upgrades.

### WP-7 — Research material mechanics

Issue: #76.

Status: `review-complete`.

Deliverables:

- Add `.txt` extraction status to SME inventory.
- Make `researcher` responsible for extraction of large local PDFs.
- Clarify IP policy: text extraction inherits the same local-use /
  paraphrase-and-cite posture as the source PDF.

## Exit criteria

- Every P0 issue above is either closed by an rc4 commit or explicitly
  downgraded by customer ruling recorded in `CUSTOMER_NOTES.md`.
- Every rc4 change has code-reviewer review before tag.
- `scripts/smoke-test.sh` passes.
- Upgrade warning paths added in rc4 have focused tests or documented
  manual verification.
- Retrofit playbook changes have at least one redaction-oriented review
  pass before tag.
- `CHANGELOG.md` contains a `v1.0.0-rc4` section naming the issue set.

Current review record: `docs/audits/v1.0.0-rc4-review.md`.
Final-readiness checklist: `docs/v1.0.0-final-checklist.md`.

## rc5 issue-boundary addendum

Issues #84 through #103 are the post-rc4 stabilization set. They do
not change the historical rc4 decision; they change the release
boundary by requiring `v1.0.0-rc5` and rc5 validation before final.

| Issue set | Disposition for rc5 | Rationale |
|---|---|---|
| #84-#86 | rc5 | Upgrade bootstrap, Codex adapter installation, and downstream exclusion fixes affect whether rc4 can be adopted safely. |
| #87-#90 | rc5 | Downstream review found local-supplement, retrofit, deliverable-shape, and documentation ambiguity that should be settled before final. |
| #91-#93 | rc5 | Final gates, customer-truth ownership language, and Codex parity checks need evidence-backed release wording. |
| #94-#96 | rc5 | Codex naming, dispatch authorization, and reasoning-effort rules are part of the dual-harness contract. |
| #97-#98 | rc5 | Tech-lead and SME local-supplement conflict handling must be unambiguous before final. |
| #99-#102 | rc5 | Framework / project path ownership, queued specialist work, completed-agent closure, and release-audit scope classification prevent downstream product work from absorbing framework maintenance. |
| #103 | rc5 | Missing Codex subagent completion/status recovery can deadlock required specialist work and must be documented before final. |

Final `v1.0.0` now depends on `v1.0.0-rc5` tagging, downstream rc5
validation, and the objective gates in
`docs/v1.0.0-final-checklist.md`.

  (#27) in rc4.
- Do not make GitHub Projects mandatory for rc4.
- Do not solve harness-level subagent budget or `SendMessage` support
  directly; record the upstream limitation and provide template-level
  mitigations.
