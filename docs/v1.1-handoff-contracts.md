# v1.1 Handoff Contracts

Implementation reference for v1.1.0. The hook infrastructure described
here is implemented. This document defines the contract surface, migration
order, and behavior of each shipped hook for enforceable machine-readable
task handoffs.

## Goal

v1.1 adds machine-readable task and handoff contracts that hooks can
validate deterministically. The contracts make the existing team model
enforceable; they do not replace it.

Baseline role model remains unchanged:

- Main session: `tech-lead`.
- Implementation: `software-engineer`.
- Review: `code-reviewer`.
- Security: `security-engineer`.
- Release: `release-engineer`.
- Claude Code and Codex are harnesses, not roles.

## Bounded Codex Mode

Add optional bounded-Codex mode only when the active handoff explicitly
permits it. Default Codex behavior remains the existing adapter model:
the top-level session is `tech-lead`, specialist work stays routed, and
top-level production authoring remains restricted.

Bounded-Codex handoffs must declare:

- Which role-owned action is permitted at the top level.
- Exact `allowed_paths` and forbidden paths.
- Required evidence before any done claim.
- Expiry or completion condition for the exception.

## Proposed Structure

Target layout:

```text
CLAUDE.md                         # Claude Code entrypoint, role contract
AGENTS.md                         # Codex adapter to the same role contract
.claude/agents/*.md               # shipped canonical role contracts
.claude/agents/*-local.md         # project-local customizations only
docs/handoffs/*.json              # durable handoff contracts
.devteam/active-handoff.json      # pointer to the active handoff
.claude/settings.json             # hook wiring and warning/enforce mode
scripts/hooks/*.py                # deterministic hook gates
```

- `docs/handoffs/<task_id>.json` — durable task contract per coherent
  task or release handoff.
- `.devteam/active-handoff.json` — pointer to the active handoff, not a
  second source of truth.
- `schemas/handoff.schema.json` — JSON Schema for required fields,
  role ownership, path scope, evidence requirements, and gate mode.
- `scripts/validate-handoff.py` — schema and repository-consistency
  validator.
- `scripts/hooks/lib/path_scope.py` — shared allowed / forbidden path
  matching and framework/product boundary checks.
- `scripts/hooks/lib/write_targets.py` — shared write-target extraction
  for hook events and shell commands.
- `scripts/hooks/lib/handoff.py` — active-handoff loading, validation,
  gate-mode handling, and evidence lookups.
- `scripts/hooks/handoff-pre-tool-gate.py` — write / command scope gate.
- `scripts/hooks/handoff-record-activity.py` — append deterministic
  activity evidence from hook events.
- `scripts/hooks/handoff-task-created-gate.py` — validate newly created
  specialist tasks against the active handoff.
- `scripts/hooks/handoff-task-completed-gate.py` — block completion
  without required independent evidence.
- `scripts/hooks/handoff-subagent-stop-gate.py` — block specialist stop
  when required return evidence is missing.
- `scripts/hooks/handoff-stop-gate.py` — block top-level stop when the
  active handoff is inconsistent or incomplete.

## Enforcement Boundaries

Hooks should enforce deterministic boundaries from the active handoff:

- No edits outside `allowed_paths`.
- Forbidden path blocks always win over broad allow rules.
- Framework/product boundary rules remain active for downstream trees.
- Customer truth stays librarian-stewarded; human approval must come
  from `CUSTOMER_NOTES.md` or another librarian-stewarded record.
- Direct `tech-lead` production authoring remains blocked unless the
  handoff grants a bounded exception.
- No done claims without independent evidence.
- No top-level stop while the active handoff is internally inconsistent,
  incomplete, or contradicts recorded gate state.

## Hard-Rule Trace Classifications

Every durable handoff contract carries a `hard_rule_traces` array. Each
entry documents which project Hard Rule the handoff touches or satisfies.

### Field shape

Each trace object has three fields (from `schemas/handoff.schema.json`
`$defs/hard_rule_trace`):

| Field | Required | Type | Purpose |
|---|---|---|---|
| `rule_id` | yes | string | Stable identifier in the form `HR-<n>` (e.g. `HR-01`). |
| `description` | yes | string | One sentence explaining how this handoff relates to the rule. |
| `source` | no | string | Canonical source document, typically `CLAUDE.md ## Hard rules`. |

### When to record a trace

A trace is required whenever the handoff's scope, gates, or evidence
requirements directly enforce, restrict, or satisfy a Hard Rule. A handoff
that does not touch a given rule omits that rule's trace.

Common triggers:

- The handoff restricts who may interact with the customer (HR-01).
- The handoff declares a safety-critical or domain-critical path that needs customer sign-off (HR-02, HR-04).
- The handoff requires a `code-reviewer` artifact before completion can be accepted (HR-03).
- The handoff gates a security-sensitive release on `security-engineer` sign-off (HR-07).
- The handoff explicitly blocks top-level `tech-lead` production authoring or grants a bounded exception (HR-08).
- The handoff declares `framework_scope: framework-maintenance` to authorize edits to framework-managed paths (HR-10).
- The handoff references the customer-question gate or `scripts/lint-questions.sh` enforcement (HR-11).

### Canonical HR index

The authoritative list is `CLAUDE.md ## Hard rules`. The stable
identifiers below map to that list by ordinal position. Use zero-padded
two-digit numbers (`HR-01` through `HR-11`) in all traces and hook messages
so lexical sort matches numeric sort.

| ID | Summary | Typical handoff impact |
|---|---|---|
| HR-01 | Only `tech-lead` interfaces with the customer. | Trace when the handoff includes a customer-question gate or restricts who may ask the customer. |
| HR-02 | No production code ships on safety-critical or domain-critical paths without customer sign-off recorded in `CUSTOMER_NOTES.md`. | Trace when the handoff covers safety-critical or domain-critical paths; `requires.human_approval` must be `true`. |
| HR-03 | No commit without `code-reviewer` review. | Trace when the handoff produces commits; `requires.review` must be `true` and a `code-reviewer` evidence gate must be present. |
| HR-04 | Changes touching safety-critical, irreversible, or customer-flagged critical logic require live customer approval — no cached approval, no agent-only path. | Trace when the handoff includes irreversible changes; `requires.human_approval` must be `true` and the evidence gate must cite a librarian-stewarded customer-truth record. |
| HR-05 | Prefer paraphrase over quotation from standards documents. | Trace when the handoff produces documentation that references SWEBOK, IEEE, or ISO sources. Semantic rule; no dedicated evidence gate. |
| HR-06 | Before escalating to `tech-lead`, check `CUSTOMER_NOTES.md` and consider whether another agent is the right addressee. | Trace when the handoff involves inter-agent routing decisions or customer-truth lookups. Semantic rule; may cite a librarian-stewarded lookup as evidence. |
| HR-07 | Releases touching authentication, authorization, secrets, PII, or network-exposed endpoints require `security-engineer` sign-off recorded in `CUSTOMER_NOTES.md`. | Trace when the handoff covers security-sensitive release scope; `requires.security_review` must be `true` and a `security-engineer` evidence gate must be present. |
| HR-08 | `tech-lead` orchestrates; it does not author production artifacts directly. | Trace when the handoff grants or restricts top-level production authoring. A bounded-Codex exception or explicit `owner_role` assignment documents the authorized exception. |
| HR-09 | Before closing a non-trivial turn, `tech-lead` runs the harness-appropriate pre-close audit. | Trace when the handoff includes a stop gate; the `handoff-stop-gate.py` hook enforces this check. |
| HR-10 | In downstream projects, keep product work separate from framework work. Do not edit framework-managed files without explicit authorization. | Trace when the handoff declares `framework_scope: framework-maintenance` to authorize framework path edits, or when it explicitly restricts such edits for product work. |
| HR-11 | Atomic customer questions: one decision axis per turn, batched internally, asked as the final line when all agents and tools are idle. | Trace when the handoff wires the customer-question gate or references `scripts/lint-questions.sh`. |

## Hard Rules Migration

`CLAUDE.md` and `AGENTS.md` remain the semantic source of truth for Hard
Rules. v1.1 should add a hard-rule migration map that classifies each
rule as one or more of: semantic-only, hook-enforceable,
evidence-gated, or customer-approval-gated.

Deterministic portions should move into hook checks or required handoff
schema fields while preserving traceability to the original Hard Rule
number. The migration must not rename or replace the source rule; each
gate should cite the source as `HR-<number>`.

Examples:

- `HR-1` only `tech-lead` talks to the customer: semantic rule plus a
  customer-question gate.
- `HR-3` no commit without `code-reviewer`: evidence-gated review
  requirement.
- `HR-7` security-sensitive releases require sign-off: security-review
  evidence gate.
- `HR-8` no `tech-lead` production authoring: existing or extended
  `tech-lead` authoring guard.
- `HR-11` atomic customer questions: `scripts/lint-questions.sh` plus
  customer-question gate checks.

Stricter `.claude/**` policy:

- Product work edits no shipped `.claude` files.
- Project customizations use `*-local.md` supplements, not edits to
  shipped role contracts.
- Framework maintenance may edit shipped `.claude` files only when the
  active handoff declares `framework_scope: framework-maintenance` and
  the required review gate is active.

## Hook Plan

| Deterministic rule | Planned hook mechanism |
|---|---|
| Edits must stay within `allowed_paths`; `forbidden_paths` wins. | `handoff-pre-tool-gate.py` on `PreToolUse` for write-capable tools and shell commands. |
| Activity evidence must be captured from hook-observable events, not prose summaries. | `handoff-record-activity.py` records normalized event, actor, target, and result. |
| Specialist task creation must cite a valid active handoff and role scope. | `handoff-task-created-gate.py` on `TaskCreated`. |
| Completion must have required independent test / review / security / customer evidence. | `handoff-task-completed-gate.py` on `TaskCompleted`. |
| Specialist stop must not discard required handoff return data. | `handoff-subagent-stop-gate.py` on `SubagentStop`. |
| Top-level stop must not leave an active handoff inconsistent or falsely complete. | `handoff-stop-gate.py` on `Stop`. |
| Bounded Codex calls must be scoped and non-transcript-based. | Codex handoff gate in the MCP / `PreToolUse` path for `mcp__codex__.*`. |

### `handoff-task-created-gate` — Task Created Gate

`scripts/hooks/handoff-task-created-gate.py`

This hook runs on the `TaskCreated` event and enforces two preconditions before a newly created task may become active work. It implements FR-021 through FR-023 and SC-007.

**US4 role.** Any tool or workflow — including Speckit-derived task output — that emits a `TaskCreated` event is subject to this gate. The gate prevents task payloads from bypassing role authority or the handoff contract by requiring both a valid owner and an explicit handoff anchor before the task is accepted.

**Check 1 — canonical owner role.** The event must carry an `owner_role` field whose value is one of the fixed canonical roster roles or matches the `sme-<slug>` pattern for dynamic SME roles. The canonical set is the same roster defined in `CLAUDE.md § Agent roster`. If `owner_role` is absent, empty, or not recognized, the gate responds with a clear reason identifying the field and listing examples of valid values.

**Check 2 — durable handoff citation.** The event must cite the active handoff by including a `handoff_task_id` or `active_handoff_task_id` field whose value matches the `task_id` on the currently active handoff. A missing field and a mismatched value are both failures, each with a distinct reason message that names both the cited value and the expected value.

**Fail-safe on load failure.** If the active handoff cannot be read or parsed, the gate does not silently pass the event. In `enforce` mode it denies; in `warn` mode it allows but emits a warning. This prevents an unreadable handoff from becoming an implicit bypass.

**Gate mode.** The hook reads `SWDT_HANDOFF_GATES` from the environment. It is silent and exits 0 when the variable is absent or set to any value other than `warn` or `enforce`. It is also silent when the incoming event is not a `TaskCreated` event.

**Output format.** When a violation is detected the hook writes a single JSON line to stdout with a `hookSpecificOutput` envelope. In `enforce` mode the `permissionDecision` is `"deny"` and a `permissionDecisionReason` field carries the reason. In `warn` mode the `permissionDecision` is `"allow"` and a `warning` field carries the reason. When all checks pass the hook produces no output.

| Condition | `enforce` | `warn` |
|---|---|---|
| `owner_role` missing or not in canonical roster | deny | allow + warning |
| `handoff_task_id` / `active_handoff_task_id` absent or empty | deny | allow + warning |
| cited handoff `task_id` does not match active handoff | deny | allow + warning |
| active handoff cannot be loaded or validated | deny | allow + warning |
| `SWDT_HANDOFF_GATES` unset or unrecognized | silent pass | silent pass |
| event is not `TaskCreated` | silent pass | silent pass |

### `handoff-record-activity` — Activity Evidence Hook

`scripts/hooks/handoff-record-activity.py`

This hook runs on `PostToolUse` (and any hook envelope event) and appends one evidence entry to the active handoff's `verification.tests` list. It produces hook-captured evidence — not worker self-attestation — so the `TaskCompleted` gate can treat the recorded activity as accepted.

**What it records.** Each entry carries:

| Field | Value |
|---|---|
| `evidence_kind` | `"accepted"` |
| `actor_role` | `"hook"` |
| `source` | Hook event name (e.g. `"PostToolUse"`) |
| `name` | Tool name plus a compact command summary (truncated to 80 characters) |
| `result` | `"passed"` |
| `timestamp` | UTC, ISO 8601 format |

All fields match `$defs/evidence_gate` in `schemas/handoff.schema.json`.

**Write safety.** The hook writes atomically: it loads the durable handoff file, appends the new entry to `verification.tests`, writes the updated JSON to a sibling `.tmp` file in the same directory, then renames the `.tmp` file over the original. On POSIX this rename is atomic. If the process is interrupted before the rename, the original handoff file is untouched. The hook appends only; it never deletes or mutates existing fields.

Evidence is written directly to the durable handoff (the file that `load_active_handoff` reads), not to a separate activity log. The gate helpers in `scripts/hooks/lib/handoff.py` read `verification.tests` directly, so a separate log would require gate changes.

**Gate mode.** The hook exits 0 silently when `SWDT_HANDOFF_GATES` is unset or set to any value other than `"warn"` or `"enforce"`. In both active modes, a load failure is non-blocking: the hook logs a warning to stderr and exits 0. Recording activity must not block a tool call; only the `TaskCompleted` gate blocks.

**No permission decision.** This hook does not produce a stdout JSON response and does not allow or block any tool. It is purely an evidence recorder.

Claude Code hook event nuance:

- `TaskCreated`, `TaskCompleted`, and `Stop` do not use matchers.
- `PreToolUse` can match `Edit`, `Write`, `MultiEdit`, `Bash`,
  `ExitPlanMode`, and `mcp__codex__.*`.

### `handoff-subagent-stop-gate` — Subagent Stop Gate

`scripts/hooks/handoff-subagent-stop-gate.py`

This hook runs on the `SubagentStop` event and ensures that a returning specialist has deposited the evidence it was responsible for on the active durable handoff before its work leg is accepted. It implements FR-027 and US5 (SC-003, SC-004).

**Purpose.** A specialist completing its leg cannot silently stop without leaving the return evidence that downstream gates — including `handoff-task-completed-gate.py` — will require. This hook catches a missing-evidence condition at the earliest possible point: the moment the subagent stops.

**Role-to-evidence mapping.** The check performed depends on the `subagent_role` field in the `SubagentStop` event:

| Returning role | Evidence checked | Handoff field consulted |
|---|---|---|
| `code-reviewer` | Review evidence present | `requires.review` + `verification.reviews` |
| `security-engineer` | Security evidence present | `requires.security_review` + `verification.security` |
| `librarian` | Human-approval / customer-truth evidence present | `requires.human_approval` + `verification.human_approval` |
| `researcher` | Human-approval / customer-truth evidence present (backward-compat: accepted alongside `librarian` for existing/historical handoff records where `actor_role: "researcher"` was recorded before the librarian split) | `requires.human_approval` + `verification.human_approval` |
| Any other role (or role not supplied) | Full `missing_evidence_gates()` check — all required gates | All `requires.*` fields |

For each mapped role, the gate checks only the evidence bucket that role owns, and only when the handoff marks that bucket as required (`true`). If the handoff does not require that bucket for the returning role, the hook passes silently. For an unrecognized or absent role, the full gate set is evaluated as a conservative fallback.

**Gate mode.** The hook reads `SWDT_HANDOFF_GATES` via the shared `resolve_gate_mode()` helper. It is silent and exits 0 when that variable is absent or set to any value other than `warn` or `enforce`. In `enforce` mode a violation is returned as a `deny` decision; in `warn` mode it is returned as an `allow` with an attached warning message. The hook is always silent when the event is not a `SubagentStop`.

**Fail-safe on load failure.** If the active handoff cannot be read or parsed, the hook does not silently pass. In `enforce` mode it denies; in `warn` mode it allows but emits a warning. An unreadable handoff is never treated as an implicit bypass.

**Output format.** When a violation is detected the hook writes a single JSON line to stdout with a `hookSpecificOutput` envelope. The reason message names the returning role and the missing evidence labels. When all checks pass or the gate is off, the hook produces no output.

| Condition | `enforce` | `warn` |
|---|---|---|
| Required evidence for the returning role is missing | deny | allow + warning |
| Role not supplied — full gate check finds missing evidence | deny | allow + warning |
| Active handoff cannot be loaded or validated | deny | allow + warning |
| `SWDT_HANDOFF_GATES` unset or unrecognized | silent pass | silent pass |
| Event is not `SubagentStop` | silent pass | silent pass |
| Required evidence is present (or not required) | silent pass | silent pass |

### `handoff-stop-gate` — Top-Level Stop Gate

`scripts/hooks/handoff-stop-gate.py`

This hook runs on the `Stop` event — the top-level (main-agent) session stop — and inspects the active handoff for three categories of bad terminal state. It implements FR-027 and US5.

**No active handoff.** When `.devteam/active-handoff.json` is absent, the session is stopping without any active handoff. The hook exits silently (clean allow, no output). A completed or cleared handoff represents the expected clean exit path; the hook has nothing to check.

**Three detected conditions.** Each violation carries a bracketed label in the reason field:

| Condition label | Trigger |
|---|---|
| `INCONSISTENT` | The active pointer is unreadable or malformed; the target file is missing or cannot be parsed; schema validation fails; required structural fields (`allowed_paths`, `forbidden_paths`) are absent; or the handoff status is neither `active` nor a completed-family value (`completed`, `done`, `closed`). |
| `INCOMPLETE` | Status is `active` and `missing_evidence_gates()` returns one or more unsatisfied required gates. The session is stopping with open evidence obligations. |
| `FALSELY_COMPLETED` | Status is `completed`, `done`, or `closed` but `missing_evidence_gates()` finds unsatisfied required gates. Completion was claimed without the required independent evidence. |

**Gate mode.** The hook reads `SWDT_HANDOFF_GATES` via the shared `resolve_gate_mode()` helper. When the variable is absent or set to any value other than `warn` or `enforce`, the hook exits silently without producing any output. In `enforce` mode a violation is returned as a `deny` decision with a `permissionDecisionReason`. In `warn` mode it is returned as an `allow` with a `warning` field. All violation reasons include the bracketed condition label.

**Scope difference from `handoff-subagent-stop-gate`.** The subagent stop gate fails safe on a missing handoff (an unexpected specialist stop) and checks only the evidence bucket for the returning role. The top-level stop gate treats a missing handoff as a clean exit and runs the full evidence check against the entire active handoff.

**Output format.** When a violation is detected the hook writes a single JSON line to stdout with a `hookSpecificOutput` envelope and `hookEventName: "Stop"`. When all checks pass or the gate is off, the hook produces no output.

| Condition | `enforce` | `warn` |
|---|---|---|
| Pointer unreadable or malformed | deny `[INCONSISTENT]` | allow + warning `[INCONSISTENT]` |
| Target file missing or unparseable | deny `[INCONSISTENT]` | allow + warning `[INCONSISTENT]` |
| Schema validation fails | deny `[INCONSISTENT]` | allow + warning `[INCONSISTENT]` |
| Structural fields missing or status not recognized | deny `[INCONSISTENT]` | allow + warning `[INCONSISTENT]` |
| Status `active` with unsatisfied evidence gates | deny `[INCOMPLETE]` | allow + warning `[INCOMPLETE]` |
| Status completed-family with unsatisfied evidence gates | deny `[FALSELY_COMPLETED]` | allow + warning `[FALSELY_COMPLETED]` |
| No active handoff pointer | silent pass | silent pass |
| All checks pass | silent pass | silent pass |
| `SWDT_HANDOFF_GATES` unset or unrecognized | silent pass | silent pass |
| Event is not `Stop` | silent pass | silent pass |

## Evidence Rules

Evidence must not be self-attested by the actor claiming completion.

- Tests: a `verification.tests` entry satisfies the test gate only when
  it carries `result: "passed"` AND `evidence_kind: "accepted"` explicitly.
  An absent `evidence_kind` does not satisfy the gate, and
  `evidence_kind: "worker_report"` never satisfies it regardless of
  `actor_role`. Worker self-attestation — including prose "tests passed"
  claims with no `evidence_kind` field — cannot satisfy the test gate.
  Hook-captured evidence written by `handoff-record-activity.py` always
  carries `evidence_kind: "accepted"` and `actor_role: "hook"`, so it
  satisfies this gate automatically.
- Code review requires a `code-reviewer` artifact (`actor_role:
  "code-reviewer"`, `result: "approved"`).
- Security review requires a `security-engineer` artifact (`actor_role:
  "security-engineer"`, `result` in `"approved"` or `"passed"`).
- Human approval requires `CUSTOMER_NOTES.md` or another
  librarian-stewarded customer-truth record (`actor_role: "librarian"`,
  `source: "CUSTOMER_NOTES.md"`, `result: "approved"`). For backward
  compatibility with handoff records created before the librarian split,
  `actor_role: "researcher"` is also accepted by the gate — mirroring
  `has_human_approval_evidence` in `scripts/hooks/lib/handoff.py`
  (`librarian` is primary; `researcher` is retained for historical records).
- Release handoffs must cite artifact lists rather than assuming a root
  `CHANGELOG.md`; downstream products may not have one.

The worker who performs a task cannot self-attest required tests,
reviews, security approval, or human approval. They may report what they
ran or observed, but the handoff only satisfies those gates through
hook-captured evidence, specialist-owned artifacts, or librarian-
stewarded customer-truth records.

## Bounded Codex Requirements

Bounded Codex mode is opt-in per handoff and must be narrow enough for
deterministic hooks to audit.

- The active handoff must set `mode.execution: bounded-codex`,
  `mode.codex_allowed: true`, and `mode.codex_server: codex`.
- Each MCP call must include `task_id`, the handoff path,
  `allowed_paths`, `forbidden_paths`, and `acceptance_criteria`.
- MCP calls must not include the full session transcript or unrelated
  project context.
- Bounded Codex does not waive role ownership, evidence, path-boundary,
  customer-truth, or `.claude/**` rules.

**Known limitation — Signal-2 exception sub-scope applies to all sessions.**
The pre-tool gate (`scripts/hooks/handoff-pre-tool-gate.py`) detects bounded-Codex
activity via two signals: (1) the event payload's `execution_mode` marker, or
(2) the active handoff simply carrying a `bounded_codex_exception` block. Signal 2
means that if a handoff carries a `bounded_codex_exception`, every write checked
against that handoff — including writes from a normal Claude Code session — is
gated against the narrower exception sub-scope (`allowed_paths`/`forbidden_paths`
intersection of handoff scope and exception scope), not the full handoff scope. A
write within the handoff's `allowed_paths` but outside the exception's `allowed_paths`
is denied even for a non-Codex session. This is an intentional fail-safe design:
the gate over-gates rather than under-gates when the scope is ambiguous. If a Claude
Code session needs broader write access concurrently, use a separate handoff without
a `bounded_codex_exception` block, or ensure the exception's `allowed_paths` covers
the full intended surface. During the warn-to-enforce rollout (`SWDT_HANDOFF_GATES=warn`),
over-gating surfaces as a warning rather than a hard block; see `test-codex-handoff-gate.sh`.

## Provider/Model Fallback Behavior

When a configured provider or model assignment is unavailable at dispatch
time, the framework records the deviation and applies a binding validity
rule before allowing work to continue.

### Model Fallback Record

A `model_fallback` object on the durable handoff captures the deviation.
All five fields are required when the object is present:

| Field | Type | Description |
|---|---|---|
| `requested_role` | string | The canonical role that was being dispatched (e.g. `software-engineer`). |
| `requested_model_class` | string | The model class preferred for that role at dispatch time. |
| `actual_model_class` | string | The model class that was actually assigned after fallback. |
| `capability_tier_comparison` | enum: `same` / `higher` / `lower` | Declares how the actual class compares to the requested class. Set by the role/model-routing owner. |
| `fallback_reason` | string | Plain-language explanation of why the preferred assignment was unavailable. |

### Validity Rule

A fallback is acceptable only when `capability_tier_comparison` is `same`
or `higher`. The phrase **"same-or-higher capability tier"** is the binding
threshold: the actual model must be at least as capable as the one
requested.

A `lower` value means the actual model is a downgrade relative to the
request. `scripts/validate-handoff.py` (`check_model_fallback_tier`) treats
this as a validation failure. Work must pause or escalate; it must not
proceed automatically at reduced capability.

There is no authoritative numeric tier ranking encoded in the repository
schemas. Validation relies entirely on the declared
`capability_tier_comparison` value. The role/model-routing owner sets this
field and is responsible for its accuracy.

### Composition with Bounded Codex

Neither bounded Codex mode nor a model fallback record may waive any of
the following:

- Role ownership — the canonical owner role on the handoff does not change.
- Path scope — `allowed_paths` and `forbidden_paths` remain binding.
- Evidence gates — all required test, review, security, and customer-truth
  gates must still be satisfied before completion is accepted.
- Customer-truth stewardship — `librarian` remains the only agent that
  records customer-truth entries in `CUSTOMER_NOTES.md`.
- Framework boundary — framework-managed paths still require
  `framework_scope: framework-maintenance` declared in the handoff.

A provider/model assignment failure must not collapse specialist-owned work
into top-level `tech-lead` production authoring. If that collapse would be
needed, a separate bounded-Codex handoff must explicitly grant the exception
before work proceeds.

## Related Workflow Integrations

v1.1 should define adjacent planning/spec tools as integrations with the
handoff model, not as bypasses around the role model.

### `llmdc`

`llmdc` is an INPUT/HELPER adapter in this framework. It is not a parallel
authority, a canonical role, or a role-roster member. Its activities may
feed planning, documentation, evidence-input, or coordination surfaces, but
they do not constitute role-owned decisions.

**What llmdc is.** An llmdc activity is a planning, documentation,
evidence, or coordination output produced by the llmdc tool. It may
prepare or summarize material that a canonical role will review and act on.
The canonical role that receives the output owns the decision and the
resulting artifact. llmdc does not stand in for that role.

**Owner.** Every llmdc activity that affects task scope, path permissions,
completion state, or evidence claims must cite the active handoff
(`docs/handoffs/<task_id>.json` as referenced by `.devteam/active-handoff.json`).
The canonical owner role declared on that handoff is the authority for
accepting or rejecting the llmdc output. An llmdc activity without a cited
active handoff is out of scope for enforcement purposes and may not affect
gate state.

**Allowed surfaces.** llmdc output may feed the following surfaces:

| Surface | Notes |
|---|---|
| Planning | Draft objective statements, candidate acceptance criteria, scope proposals — routed to `tech-lead` or `project-manager` for ownership review. |
| Documentation | Draft prose, section outlines, reference material — routed to `tech-writer` for ownership review. |
| Evidence-input | Tool-call summaries, activity logs, rerun summaries — appended as input to the handoff's `activity` array for review by the owning role; not treated as accepted evidence until the gate hook or reviewer accepts them. |
| Coordination | Task comments, status summaries, inter-role routing notes — visible to the receiving role; not binding until the receiving role acts. |

**Evidence limits (critical).** These prohibitions are absolute and cannot
be waived by a handoff declaration:

- llmdc CANNOT mark role-owned work complete. Completion requires all
  required evidence gates to be satisfied through hook-captured activity,
  specialist review artifacts, or librarian-stewarded customer records.
- llmdc CANNOT approve evidence gates. Gate approval is performed by the
  owning canonical role using the mechanisms defined in the Evidence Rules
  section of this document.
- llmdc CANNOT record customer truth independently. Only `librarian`
  stewards `CUSTOMER_NOTES.md`. Only `tech-lead` interfaces with the
  customer. An llmdc output that purports to record a customer answer or
  customer approval is not valid and must not enter the handoff's
  `verification` or `completion` records without librarian stewardship.

llmdc output is input-only until a canonical role accepts it through the
owning handoff's evidence gates. The worker who submits llmdc output cannot
also serve as the accepting authority for that same output.

### Speckit

Speckit should be modeled as a planning/spec workflow integration whose
outputs feed canonical roles and handoffs.

- Use `speckit-specify` to turn customer-scoped feature intent into a
  draft spec, subject to `tech-lead` customer-question rules for any
  missing policy, priority, or acceptance decision.
- Use `speckit-plan` for design and implementation planning inputs that
  route to `architect`, `project-manager`, and affected specialist roles
  for ownership review.
- Use `speckit-tasks` to derive candidate task breakdowns, then map each
  task to a canonical owner role and, when active work starts, to a
  `docs/handoffs/*.json` contract.
- Use `speckit-implement` only through normal role routing and active
  handoff scope; it must not bypass bounded-Codex limits, specialist
  ownership, path gates, review gates, or customer-truth stewardship.
- Use `speckit-analyze` and `speckit-checklist` as planning-quality and
  readiness checks; their findings can become acceptance criteria,
  checklist items, or review inputs, but not final evidence by themselves.
- Define how Speckit artifacts map to handoff fields such as objective,
  allowed paths, acceptance criteria, required reviews, and migration /
  release notes, while keeping `docs/handoffs/*.json` as the durable task
  contract for active work.
- Keep `tech-lead` as the only customer interface: Speckit may surface
  questions, but queued customer questions, final wording, and batching
  remain governed by the existing customer-question gate.

#### Speckit Artifact Mapping

Speckit artifacts map to handoff fields as follows. The mapping covers how each
Speckit artifact type relates to fields on a durable handoff contract, the role
owner who accepts the output, and the source-authority class that governs how
the artifact may be used.

**Artifact-to-field mapping**

| Speckit artifact | Primary handoff field(s) | Accepting role(s) | Source authority |
|---|---|---|---|
| `spec.md` (feature spec) | `objective`, `acceptance_criteria`, requirements narrative | `tech-lead`, `architect`, `project-manager` | Canonical input — closest to customer intent; treated as the authoritative scope statement until explicitly superseded by a role-owned decision. |
| `plan.md` (implementation plan) | `allowed_paths`, `forbidden_paths`, `framework_scope`, `owner_role`, `review_roles` candidates | `architect`, `project-manager`, `tech-lead` | Generated/planning artifact — not a durable handoff; feeds handoff field drafts but requires role-acceptance before the resulting handoff is active. |
| `data-model.md` | `acceptance_criteria`, `requires` (evidence gate types), domain entity definitions | `architect`, `software-engineer` | Generated/planning artifact — informs schema and gate design; not a requirement record itself. |
| `contracts/` (hook-events, handoff-contract) | `requires`, `verification` structure, gate-mode declarations | `software-engineer`, `architect` | Generated/planning artifact — treated as design input; the shipped `schemas/handoff.schema.json` is canonical. |
| `tasks.md` (task list) | `task_id`, `owner_role`, `objective` per task, candidate `acceptance_criteria` | `tech-lead`, owning specialist role | Generated/planning artifact — each candidate task row maps to one `docs/handoffs/<task_id>.json` contract once the owning role accepts it. |
| Checklist output (`speckit-checklist`) | `acceptance_criteria`, checklist items fed to `requires` | Owning specialist role | Generated/planning artifact — findings may become acceptance criteria or review inputs; may not count as accepted evidence. |
| Analyze output (`speckit-analyze`) | `acceptance_criteria`, risk notes, readiness inputs | `tech-lead`, `qa-engineer`, `architect` | Generated/planning artifact — readiness observations feed gate design and review scope; do not constitute final evidence by themselves. |

**Role-owner mapping (input-only rule)**

Speckit output feeds role-owned work and durable-handoff creation but is INPUT ONLY.
Analysis, checklist, and task output from Speckit does not become accepted until
the owning canonical role reviews it and either incorporates it into a durable
handoff or records acceptance through the handoff's evidence gates. Speckit does
not approve gates, does not record customer truth, and does not bypass role authority.

**Source-authority classification**

Following the framing from `plan.md`'s own constitution check:

- `spec.md` — canonical input. It is the closest artifact to customer intent and
  is treated as authoritative scope until a role-owned decision explicitly supersedes it.
- `plan.md`, `data-model.md`, `contracts/`, `tasks.md`, checklist output, analyze
  output — generated/planning artifacts. They are produced by Speckit orchestration
  and are valid inputs to role-owned work, but they do not govern
  `CLAUDE.md`, `AGENTS.md`, role files, or the shipped schema.

**Schema cross-reference**

The `external_tool_activity_ref` definition in `schemas/handoff.schema.json`
(`tool_type: "speckit"`) carries three Speckit-specific fields:

| Schema field | Purpose |
|---|---|
| `speckit_artifact_type` | Labels the artifact (e.g. `task-scaffold`, `acceptance-checklist`, `feature-spec`). |
| `speckit_feature_path` | Records the feature path within the Speckit artifact. |
| `speckit_source_authority` | Classifies authority as `input-only` or `pending-role-acceptance`. |

These fields appear only in the `external_tool_activity` array on a handoff —
a structurally separate location from `verification.*` and `completion.evidence`.
No field in `external_tool_activity` is read by evidence helpers as accepted evidence.
A `speckit_source_authority` value of `pending-role-acceptance` signals that the
owning role has not yet reviewed the Speckit output; `input-only` signals it may
be used as background context. Neither value permits the artifact to satisfy an
evidence gate on its own.

## Release Handoff Notes

### Release-Readiness Artifact List

A release handoff must reference a project-specific artifact list before
readiness can be accepted. The list enumerates the exact artifacts required
for that release across every required evidence bucket. Each release handoff
MUST identify concrete file paths or record references for each of the
following types: release notes, checklist, review, security, test, migration, and customer-approval
artifacts.

No artifact in the list may be assumed from a conventional project layout.
In particular, the list MUST be project-specific — no root `CHANGELOG.md`
is assumed to exist. Downstream projects supply their own release notes
location; the handoff cites it explicitly. A release readiness review that
cannot resolve every artifact reference to an actual project-specific path
or record is incomplete and must not advance to enforce mode.

### Relationship to Warning-to-Enforce Readiness

Enforce mode for release gates may be enabled only after warning-mode
smoke evidence shows zero unresolved false positives across the full
clarified workflow set: handoff create/update, allowed edit, forbidden
edit, evidence acceptance, evidence rejection, bounded Codex, and model
fallback. The artifact list check is part of that readiness baseline.
A release that lacks a resolved artifact list contributes an unresolved
gap and keeps the gates in warning mode until the gap is closed.
